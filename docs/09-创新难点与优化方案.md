# 创新点、难点、经典问题与逐步优化方案

> 在前 10 轮迭代基础上，进一步深挖创新方向、经典难题、每步可做的优化  
> 面向：多机器人协同操作（RocoBench + Ollama LLM）

---

## 一、经典问题与难点（尚未完美解决的问题）

### 1.1 LLM 空间推理不可靠 ⭐⭐⭐⭐⭐ 最难

**现象**：LLM 没有真正的 3D 空间理解能力。它看到 `rob0 at (0.5, 0.3, 0.2)` 时，无法真正判断"这个位置离目标有多远""两个机器人会不会碰撞"。

**当前方案的局限**：我们依赖 ActionFilter 过滤明显不合法的动作，但无法过滤"法但蠢"的动作（如绕远路）。

**解决方向**：
- **空间离散化**：把连续坐标映射为离散区域（"左上区""右下区"），LLM 更擅长离散概念
- **距离注入**：在 Prompt 中显式计算并注入"rob0 距离 red_cube 0.3m，rob1 距离 red_cube 0.8m"
- **碰撞预测注入**：预先计算可能的碰撞并注入 Prompt："警告：rob0 和 rob1 的路径将在点 (0.5, 0.5) 交叉"

---

### 1.2 长程任务中 LLM "遗忘"早期约束 ⭐⭐⭐⭐

**现象**：Move Rope 任务中，LLM 在第 1 步规划了"两机器人同步举起"，到第 5 步时"忘了"绳子还在空中，输出了矛盾的动作。

**根本原因**：Transformer 的注意力机制在长上下文中会稀释早期信息。

**解决方向**：
- **约束持久化**：把关键约束（"绳子必须保持高度>1.0m"）作为"sticky note"在每个 Prompt 中重复
- **状态摘要**：每 N 步让 LLM 总结当前状态和剩余目标，用摘要替代原始历史
- **关键帧回溯**：让 LLM 在每步开头强制回顾任务总目标

---

### 1.3 多机器人"死锁"问题 ⭐⭐⭐⭐

**现象**：rob0 等 rob1 先动，rob1 等 rob0 先动 → 两个都 WAIT → 超时。

**根本原因**：LLM 没有全局视角，两个独立的 LLM 调用（dialog 模式）各自做局部最优决策。

**解决方向**：
- **死锁检测器**：检测到两个机器人连续 N 步都在 WAIT → 触发"破冰"机制（随机选一个机器人先动）
- **全局锁**：在 plan 模式下，一个 LLM 调用同时规划所有机器人动作（自然避免死锁）
- **活锁检测**：与死锁不同，活锁是机器人一直在动但没进展（来回移动）

---

### 1.4 Prompt 对模型敏感度过高 ⭐⭐⭐

**现象**：为 qwen2.5:7b 优化的 Prompt 在 llama3.3:70b 上表现不同，甚至更差。

**根本原因**：不同模型对 Prompt 格式、语言风格、示例数量的敏感度不同。

**解决方向**：
- **模型自适应 Prompt**：在 Prompt 开头加入模型特定的 meta-instruction（如 "You are Llama, a helpful assistant..." vs "You are Qwen..."）
- **Prompt 版本管理**：维护"qwen 版 Prompt"和"llama 版 Prompt"两套
- **自动适配**：用一个小测试（"输出 'model=qwen' 或 'model=llama'"）判断当前模型，动态选择 Prompt 模板

---

### 1.5 动作粒度不匹配 ⭐⭐⭐

**现象**：LLM 输出的"高层动作"（"抓起杯子"）和仿真环境需要的"底层动作"（"移动夹爪到(x,y,z)，闭合夹爪，抬起..."）之间存在 gap。

**当前方案**：RocoBench 框架已经做了这个映射，但有时候 LLM 不理解为啥一个"简单"动作需要那么多步。

**解决方向**：
- **动作耗时预估**：在 Prompt 中注入每个动作的预估执行时间，让 LLM 做时间规划
- **宏动作**：把常见动作序列封装为"宏"（如 `CLEAN_CUBE(x)` = `MOVE x → SWEEP x → DUMP`），减少 LLM 需要规划的步数

## 二、有价值的创新想法（值得深入探索）

### 创新 A：分层 LLM 架构 —— 大模型规划 + 小模型执行 ⭐⭐⭐⭐⭐

**核心思想**：
- 70B 模型每 N 步做一次"高层规划"（"接下来 5 步的目标是清扫红蓝两个方块"）
- 7B 模型每步做"底层执行"（"当前应该 MOVE 还是 SWEEP？"）
- 7B 的 Prompt 中注入 70B 的规划结果作为约束

**为什么好**：
- 70B 只在关键节点调用，减少 80% 的 70B 推理时间
- 7B 推理快（~1s vs ~5s），总时间反而更短
- 70B 负责"战略"（需要强推理），7B 负责"战术"（只需要局部决策）

**伪代码**：
```python
if step_count % 5 == 0 or task_stage_changed:
    high_level_plan = llm_70b.chat(strategic_prompt)
    
low_level_prompt = f"""
高层规划（来自70B）：
{high_level_plan}

当前状态：
{observation}

请选择下一步动作（只能从可用动作列表中选）：
{valid_actions}
"""
action = llm_7b.chat(low_level_prompt)
```

### 创新 B：反事实推理 —— "如果当时选了另一个动作会怎样？" ⭐⭐⭐⭐

**核心思想**：
- 任务失败后，不只是反思"为什么失败"
- 还要问**反事实问题**："如果第 3 步我们选了 rob1: WAIT 而不是 rob1: MOVE，结果会不同吗？"
- 把反事实推理结果存入经验记忆库

**为什么好**：
- 这是人类专家分析失败的核心方法
- 可以从一次失败中提取多条教训（不只是"不要做X"，还有"应该做Y"）
- 报告里展示反事实推理非常有说服力

**Prompt 设计**：
```text
任务失败了。以下是完整执行记录：
[步骤1] rob0: MOVE red_cube → 成功
[步骤2] rob1: MOVE red_cube → 成功
[步骤3] rob0: SWEEP red_cube → 成功
[步骤4] rob1: DUMP → 失败（碰撞）

请进行反事实推理：
1. 如果步骤3改为 rob0: WAIT，rob1 先 DUMP，结果会怎样？
2. 如果步骤2改为 rob1: WAIT，结果会怎样？
3. 从这些反事实中，我们能学到什么通用规则？
```

### 创新 C：LLM 驱动的课程学习 —— 从简单任务迁移到困难任务 ⭐⭐⭐⭐

**核心思想**：
- 先在 Sweep Floor（最简单）上积累成功案例
- 把 Sweep Floor 的"清扫→倒入"模式迁移到 Pack Grocery（"抓起→放入箱子"）
- 用 TaskDecomposer 找到跨任务的共同子模式

**为什么好**：
- 6 个任务不是孤立的，它们共享底层操作模式
- 经验可以在任务间迁移（Sweep Floor 的"等待协调"经验对 Move Rope 有用）
- 越简单的任务成功后，经验越多，困难任务表现得越好

**实现**：
```python

    # 提取模式（如"等待对方完成后再操作"）
    # 注入到 target_task 的 Prompt 中
    return build_cross_task_prompt(target_task, source_patterns)
```

### 创新 D：动作空间自动发现 ⭐⭐⭐

**核心思想**：
- 目前 ActionFilter 的动作空间是**手工预定义**的
- 但有时候 LLM 会输出"看起来合理但不在预定义列表中的动作"
- 能不能让系统**从 LLM 的输出中自动发现**新的有效动作？

**实现方法**：
1. 记录所有 LLM 输出的动作（包括被 ActionFilter 拦截的）
2. 统计哪些"被拦截的动作"如果执行了反而会成功
3. 如果某个被拦截动作出现频率高且逻辑合理，自动加入动作空间

**示例**：
```python
# LLM输出: rob0: PUSH red_cube （被拦截，因为PUSH不在预定义中）
# 但 PUSH 其实是合理的（在狭小空间中推进物体）
# → 自动将 PUSH 加入动作空间
```

### 创新 E：Prompt 自动变异与进化 ⭐⭐⭐⭐

**核心思想**：
- 用另一个 LLM（或同一个 LLM 以不同 temperature）**自动生成 Prompt 的变体**
- 对每个变体运行 evaluator.py
- 保留成功率最高的变体
- 重复 N 代

**为什么好**：
- 完全自动化，不需要人工试 Prompt
- 可以探索人类想不到的 Prompt 形式
- 配合经验记忆，越跑越好

**伪代码**：
```python
def evolve_prompt(base_prompt, n_generations=5, population_size=10):
    population = [mutate_prompt(base_prompt) for _ in range(population_size)]
    
    for gen in range(n_generations):
        scores = []
        for prompt in population:
            score = evaluate_prompt(prompt, task="sweep_floor", runs=5)
            scores.append((score, prompt))
        
        # 保留 top 3
        scores.sort(reverse=True)
        survivors = [p for _, p in scores[:3]]
        
        # 变异生成下一代
        population = survivors
        for _ in range(population_size - len(survivors)):
            parent = random.choice(survivors)
            child = mutate_prompt(parent)  # LLM驱动的变异
            population.append(child)
    
    return max(population, key=lambda p: evaluate_prompt(p))
```

### 创新 F：不确定性感知规划 ⭐⭐⭐

**核心思想**：
- LLM 不只是输出"最佳动作"，还要输出**对每个动作的信心度**
- 如果信心度低（< 0.6），触发额外的验证或保守策略
- 信心度高时可以直接执行，节省时间

**Prompt 设计**：
```text
请为以下候选动作评分（1-5分信心度），并只选择信心度≥3的动作。

候选动作：
0. rob0: MOVE red_cube
1. rob1: SWEEP red_cube
2. rob0: WAIT

输出格式：
<SCORES>
0: 4, 1: 2, 2: 5
</SCORES>
<CHOICE>
0
</CHOICE>
```

### 创新 G：人机协作接口（Bonus 选项）⭐⭐⭐⭐

**核心思想**：
- 在关键决策点（信心度低的动作、检测到循环时、连续失败时），**暂停并向人类求助**
- 人类给出高层建议（"试试先清理蓝色方块"），LLM 将其翻译为具体动作

**为什么出彩**：
- 这是 RocoBench 原论文的 real-world 实验方向
- "Human-in-the-loop"是具身智能的热门话题
- 即使人不在，也可以用另一个 LLM 模拟"人类建议"

## 三、每一步可以做的优化（按执行流程展开）

### 步骤 1：接收 observation → 格式化为文本

| 优化点 | 当前方案 | 改进方案 | 预期收益 |
|--------|----------|----------|:---:|
| 坐标离散化 | 直接传入浮点坐标 | 映射为空间区域描述（"左上区"） | LLM 理解更准确 |
| 关键信息高亮 | 所有 observation 字段平铺 | 用 🔴🟢 标注关键变化 | 减少 LLM 遗漏 |
| 相对位置计算 | 只给出绝对坐标 | 额外计算并注入相对距离 | 改善空间推理 |
| 物体状态摘要 | 列出所有物体状态 | 只列出"发生了变化的"物体 | 减少 Prompt 长度 |

### 步骤 2：构建 Prompt

| 优化点 | 当前方案 | 改进方案 | 预期收益 |
|--------|----------|----------|:---:|
| Prompt 长度控制 | 固定注入所有模块的输出 | 根据任务阶段动态裁剪 | 减少 token 消耗 |
| 示例选择 | 按时间取最近 N 个 | 按相似度检索最相关示例 | Few-Shot 更精准 |
| 约束优先级 | 所有约束平等 | 核心约束（动作合法性）> 建议约束（经验提示） | 关键约束不被稀释 |
| 格式自适应 | 统一 <THINK>/<ACTIONS> 格式 | 根据模型类型微调格式模板 | 跨模型兼容性 |

### 步骤 3：LLM 推理

| 优化点 | 当前方案 | 改进方案 | 预期收益 |
|--------|----------|----------|:---:|
| 温度调度 | 全局固定 0.1 | 关键步骤用 0.0，探索步骤用 0.3 | 平衡稳定与多样 |
| 模型分层 | 全用 70B | 简单步骤用 7B，复杂步骤用 70B | 减少 60% 推理时间 |
| 推理缓存 | 无 | 相似状态复用之前 LLM 回复 | 减少重复调用 |
| 流式输出 | 等完整回复 | 边生成边解析（提前截断） | 减少等待时间 |

### 步骤 4：解析输出 → 动作

| 优化点 | 当前方案 | 改进方案 | 预期收益 |
|--------|----------|----------|:---:|
| 多级回退 | 解析失败→修复→失败→默认WAIT | 增加"启发式规则回退"层 | 更多情况能恢复 |
| 动作签名验证 | 简单字符串匹配 | 用正则 + 动作空间校验 | 更严格的合法性 |
| 时序一致性 | 无检查 | 检查新动作与上一步是否逻辑连续 | 减少跳跃式规划 |
| 异常动作标记 | 无 | 标记"可疑动作"（如连续3次WAIT） | 提前预警 |

### 步骤 5：执行动作 → 获取反馈

| 优化点 | 当前方案 | 改进方案 | 预期收益 |
|--------|----------|----------|:---:|
| 超时预警 | 10分钟硬超时 | 8分钟时主动简化策略 | 减少超时失败 |
| 并行监控 | 无 | 执行时同时运行碰撞预测 | 提前发现风险 |
| 部分成功记录 | 只有成功/失败 | 记录"部分成功"（到了一半） | 更细粒度的反馈 |
| 执行时间记录 | 无 | 记录每步耗时，识别瓶颈 | 指导优化方向 |

### 步骤 6：反思与重规划

| 优化点 | 当前方案 | 改进方案 | 预期收益 |
|--------|----------|----------|:---:|
| 失败聚类 | 每次独立反思 | 将相似失败聚类，批量分析 | 发现系统性缺陷 |
| 成功也反思 | 只反思失败 | 成功时也分析"为什么成功" | 强化正确模式 |
| 多假设推理 | 只给一个失败原因 | 给 3 个可能原因+概率 | 更全面的分析 |
| 反思缓存 | 无 | 同类失败不重复反思 | 减少 LLM 调用 |

## 四、可以写在报告里的"尚未解决但有价值的开放问题"

这些问题目前还没解决，但作为"未来工作"写在报告里非常加分：

1. **如何让 LLM 真正理解 3D 空间？** —— 当前所有的 LLM 都没有空间理解能力，只能靠文本描述近似。是否有办法将空间关系编码为 LLM 能理解的格式？

2. **多机器人规划的最优通信频率是多少？** —— 每一步都通信太慢，从不通信无法协同。是否存在一个最优频率？

3. **LLM 的"规划深度"是否有上限？** —— 7B 模型似乎只能规划 3-5 步，70B 能到 8-10 步。这个上限是由什么决定的？

4. **能否用更小的模型达到相同效果？** —— 如果 0.5B 模型配合精心设计的 Prompt 和动作过滤，能接近 7B 的效果吗？

5. **不同仿真环境之间的迁移能力？** —— 在 RocoBench 上学到的策略，能迁移到真实机器人吗？需要什么适配？
