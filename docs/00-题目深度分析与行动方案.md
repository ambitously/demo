## 一、题目核心要素提取

### 1.1 题目信息一览表

| 维度 | 内容 |
|------|------|

| **核心目标** | 接入大模型（≤80B），实现多机器人协同对话/规划，完成操作任务，尽可能提高成功率 |

### 2.4 `comm_mode` 两种模式对比

| 特性 | `dialog`（多智能体对话式） | `plan`（集中式规划式） |
|------|---------------------------|----------------------|
| **LLM 调用方式** | 每个机器人独立调用 LLM，可以相互发送消息 | 一个 LLM 调用生成所有机器人的动作计划 |
| **优势** | 更符合自然协作模式，机器人可以"讨论"和"协商" | 全局视角，避免冲突，一次生成计划 |
| **劣势** | 多次 LLM 调用，耗时较长；消息传递可能混乱 | 单次调用，缺乏实时反馈和纠错能力 |
| **适用场景** | 需要频繁协商的任务（如 Pack Grocery 避障） | 任务明确、步骤固定的任务（如 Sort Cubes） |
| **入门建议** | 先跑通 **plan** 模式（更简单），再尝试 dialog 模式 | ✅ **推荐入门** |

## 三、6 个任务逐个分析

### 3.1 任务总览表

| 任务名称 | 机器人数量 | 协作类型 | 核心动作 | 难度 | 备注 |
|----------|-----------|---------|---------|------|------|
| **Sweep Floor** | 2 | 分工协作：一个扫 + 一个拿垃圾桶，最后倾倒 | MOVE, SWEEP, WAIT, DUMP | ⭐ 简单 | **推荐首个入手任务** |
| **Sort Cubes** | 2 | 空间分工：各自负责对应面板，互相传递方块 | 移动 + 放置 + 传递 | ⭐⭐ 中等 | 涉及区域限制和物品传递 |
| **Pack Grocery** | 2 | 路径协调：需避免碰撞 | 抓取 + 移动 + 放置 | ⭐⭐ 中等 | 路径规划是关键 |
| **Make Sandwich** | 2 | 顺序协作：轮流按正确顺序堆叠原料 | 选择原料 + 抓取 + 堆叠 | ⭐⭐⭐ 较难 | 对顺序严格要求 |
| **Move Rope** | 2 | 同步协作：一起举起绳子过墙 | 抓取 + 同步移动 + 放置 | ⭐⭐⭐ 较难 | 需要精确同步，避免碰撞 |
| **Arrange Cabinet** | 3 | 三人协作：两人开门 + 一人取杯放杯 | 保持门开 + 抓取 + 放置 | ⭐⭐⭐⭐ 最难 | 3个机器人，协作复杂度高 |

### 3.2 各任务详细分析

#### 3.2.1 Sweep Floor（清扫地面）⭐ — **推荐首选练手任务**

| 维度 | 详情 |
|------|------|
| **参与机器人** | 2 个 |
| **协作模式** | 分工 + 顺序协作：Robot A 用扫帚扫方块到簸箕，Robot B 拿垃圾桶，最后 Robot B 把簸箕内容倒入垃圾桶 |
| **动作空间** | `MOVE [target]` 移动到目标物体；`SWEEP [target]` 扫目标到簸箕；`WAIT` 等待另一个机器人完成；`DUMP` 倒入垃圾桶 |
| **观测空间** | 方块位置（桌上/簸箕里/垃圾桶里）、机器人夹爪 3D 位置 |
| **难点** | LLM 需要理解"等对方扫完再倒"的时序依赖 |
| **易错点** | 机器人忘记 WAIT 导致动作错序；LLM 返回的动作名拼写错误 |
| **建议策略** | 用 plan 模式 + 强约束 Prompt（明确步骤顺序）；这是入门最好任务 |

#### 3.2.2 Sort Cubes（分类立方体）⭐⭐

| 维度 | 详情 |
|------|------|
| **参与机器人** | 2 个 |
| **协作模式** | 空间分工 + 互助：各机器人有自己负责的区域面板，超出自己范围的方块需要对方帮忙传递 |
| **动作空间** | 移动 + 拾取 + 放置 + 传递 |
| **难点** | 判断"这个方块该归谁处理"需要空间推理；传递时需要协调握手区域 |
| **易错点** | 机器人越界操作（违反区域限制规则）；传递失败 |
| **建议策略** | Prompt 中明确描述各机器人的管辖区域；加入"检查方块颜色 → 判断归属"的逻辑 |

#### 3.2.3 Pack Grocery（打包杂货）⭐⭐

| 维度 | 详情 |
|------|------|
| **参与机器人** | 2 个 |
| **协作模式** | 路径协调：两个机器人同时在不同区域抓取杂货放到垃圾桶，需要避免碰撞 |
| **动作空间** | 抓取 + 移动 + 放置 |
| **难点** | 两个机器人运动路径可能交叉，需要协调先后顺序 |
| **易错点** | 碰撞导致物体掉落；两个机器人抢同一个物品 |
| **建议策略** | dialog 模式更适合（协商避免碰撞）；也可以 plan 模式中显式规划路径顺序 |

#### 3.2.4 Make Sandwich（做三明治）⭐⭐⭐

| 维度 | 详情 |
|------|------|
| **参与机器人** | 2 个 |
| **协作模式** | 顺序协作：两个机器人轮流按正确顺序（面包→配料→面包）堆叠，每个机器人用不同配料 |
| **动作空间** | 选择物品 + 抓取 + 堆叠（对顺序敏感） |
| **难点** | 严格顺序约束（下层面包→配料→上层面包）；双方配料不冲突 |
| **易错点** | 顺序搞反（先放配料再放面包）；同时操作导致堆叠失败 |
| **建议策略** | Prompt 中显式列出正确顺序（1→2→3→4）；加入"检查当前堆叠状态"的反馈机制 |

#### 3.2.5 Move Rope（移动绳子）⭐⭐⭐

| 维度 | 详情 |
|------|------|
| **参与机器人** | 2 个 |
| **协作模式** | 同步物理协作：两个机器人各抓绳子一端，同步举起、越过墙壁、放入凹槽 |
| **动作空间** | 抓取 + 同步举升 + 同步移动 + 放置 |
| **难点** | 需要精确的物理同步（一个快一个慢会导致绳子掉落） |
| **易错点** | 抓手碰撞（两个夹爪靠太近）；举升不同步导致绳子滑落；放入凹槽不准 |
| **建议策略** | plan 模式更合适（精确控制同步）；Prompt 中强调"同时""同步"；可能需要 retry 机制 |

#### 3.2.6 Arrange Cabinet（整理柜子）⭐⭐⭐⭐ — **最难任务**

| 维度 | 详情 |
|------|------|
| **参与机器人** | 3 个（Robot A、B 各开一扇柜门并保持，Robot C 取杯子放到杯垫） |
| **协作模式** | 三人分工 + 依赖：Robot C 的操作必须等到 A 和 B 把门打开并保持 |
| **动作空间** | 开门 + 保持 + 抓取 + 放置 |
| **难点** | 3 个机器人协调复杂度指数级上升；Robot A/B 需要"保持"动作（持续占用）；Robot C 需要避开两扇门的空间 |
| **易错点** | A/B 开门角度不够；A/B 在 C 操作时松手导致门关上；C 撞到门或夹爪；3 个机器人动作时序混乱 |
| **建议策略** | plan 模式 + 分阶段 Prompt（阶段1：开门并保持 → 阶段2：C 操作 → 阶段3：关门）；加入状态检查机制 |

### 3.3 任务入手优先级

```
Sweep Floor（先练手）
    │
    ▼
Sort Cubes / Pack Grocery（二选一，理解不同协作模式）
    │
    ▼
Make Sandwich / Move Rope（难度提升，加入状态检查）
    │
    ▼
Arrange Cabinet（最终挑战，3 机器人协调）
```

### Phase 2：Prompt 优化（约 1-2 天）

#### 目标：通过 Prompt Engineering 大幅提升成功率

#### 2.1 Sweep Floor 入手（最简单任务，快速验证 Prompt 效果）

```
1. 读原始 Prompt
   - 定位 prompter 文件中的 system_prompt 和 user_prompt
   - 理解当前 Prompt 给了 LLM 什么信息
   
2. 优化方向：
   a) 明确输出格式约束
      示例: "You MUST output actions in the format: ROBOT X: ACTION [ARGS]"
   
   b) 明确任务步骤（Few-shot / Chain-of-Thought）
      示例: "Step 1: Robot 1 moves to the broom.
            Step 2: Robot 1 sweeps cube into dustpan.
            Step 3: Robot 2 waits for Robot 1 to finish.
            Step 4: Robot 2 dumps dustpan into trash."
   
   c) 添加错误预防规则
      示例: "IMPORTANT: Robot must WAIT before the other robot finishes its task."
   
   d) 添加角色定义
      示例: "Robot 1 = Sweeper (has broom). Robot 2 = Trash Collector (has dustpan/trash can)."
   
   e) 添加空间信息描述
      示例: "The cube is at position [x, y, z]. The dustpan is at [x, y, z]."

3. A/B 测试
   - 每次改一个 prompt 变量
   - 用 evaluator.py 跑 10 次看成功率变化
   - 保留有效的改动
```

#### 2.2 推广到其他任务

```
对于每个任务，分析其特有的难点，在 Prompt 中加入针对性信息：

Sort Cubes:
  - 描述各机器人管辖区域边界
  - 加入颜色→归属的判断规则

Pack Grocery:
  - 加入路径规划的优先级规则（谁先走）
  - 检测潜在碰撞区域

Make Sandwich:
  - 显式列出完整步骤顺序
  - 每步前检查当前堆叠状态

Move Rope:
  - 强调"同时"和"同步"
  - 明确两个夹爪的安全距离

Arrange Cabinet:
  - 分阶段 Prompt（开门→取放→关门）
  - 定义"保持"动作的持续条件
```

### Phase 4：创新改进（约 2-3 天）

#### 目标：选择 1-2 个创新点实现，拉开差距

#### 创新方向 A：多轮反思与纠错机制 ⭐⭐⭐（推荐）

```
核心思路：LLM 第一次生成的计划不是最终方案，执行后检查状态，
         如果不符合预期，重新生成计划。

实现步骤：
1. 在每一轮动作执行后，获取环境状态
2. 对比预期状态和实际状态
3. 如果有偏差，将偏差信息作为反馈输入 LLM，重新生成计划
4. 示例 Prompt:
   "Previous plan failed because: {error_reason}.
    Current state: {state}.
    Please generate a corrected plan."

优点：实现简单，效果显著
缺点：增加 LLM 调用次数 → 增加时间，需控制在 10 分钟内
```

#### 创新方向 B：少样本示例（Few-shot）增强 ⭐⭐

```
核心思路：在 Prompt 中加入成功的示例（任务描述 + 理想动作序列），
         让 LLM 有模板可参考。

实现步骤：
1. 手工为每个任务编写 1-2 个成功示例
2. 示例格式: "Example Task: {description} → Plan: {action_sequence}"
3. 加入 system prompt，放在任务描述之前

优点：对 7B 模型尤其有效，不会增加运行时间
缺点：需要人工编写示例，泛化性有限
```

#### 创新方向 C：层级规划（Hierarchical Planning）⭐⭐⭐

```
核心思路：先生成高层宏观计划，再逐步细化到具体动作。

实现步骤：
1. 第一次 LLM 调用: 生成高层步骤（如 "Phase 1: 清扫桌面"）
2. 第二次 LLM 调用: 对每个阶段细化具体动作
3. 可以不同阶段用不同的 prompt 策略

优点：复杂任务（Arrange Cabinet）效果显著
缺点：多次 LLM 调用增加耗时
```

#### 创新方向 D：碰撞预测与路径协商 ⭐⭐

```
核心思路：在 plan/dialog 中显式预测碰撞并协商绕行。

实现步骤：
1. 在执行前，让 LLM 预测两个机器人的运动轨迹
2. 如果轨迹交叉，生成绕行方案
3. 可以加入"优先级"机制（某个机器人先走，另一个等待）

优点：对 Pack Grocery 和 Move Rope 特别有效
缺点：Prompt 需要包含空间坐标信息
```

#### 创新方向 E：动作验证与格式纠错 ⭐

```
核心思路：LLM 输出的动作可能格式不对（拼写错误、参数错误），
         用规则或二次 LLM 调用进行验证和修正。

实现步骤：
1. 定义合法的动作空间
2. LLM 输出后，用正则匹配检查格式
3. 如果格式错误，将错误信息反馈给 LLM 重新生成
4. 或直接用规则映射（如 "swep" → "SWEEP"）

优点：实现简单，能解决大量低级错误
缺点：只是兜底策略，不解决根本问题
```

#### 推荐创新组合方案

| 方案 | 创新点组合 | 难度 | 预期效果 | 时间 |
|------|-----------|------|---------|------|
| **快速出方案** | A（反思） + E（格式纠错） | ⭐⭐ | 中等 | 1.5 天 |
| **平衡方案** | B（Few-shot） + A（反思） + E（纠错） | ⭐⭐⭐ | 较好 | 2 天 |
| **冲高方案** | C（层级规划） + A（反思） + D（碰撞预测） | ⭐⭐⭐⭐ | 最佳 | 3 天 |

## 五、关键风险与对策

| 风险 | 影响 | 概率 | 对策 |
|------|------|------|------|
| **Ollama 模型下载慢/失败** | Phase 0 卡死 | 中 | 在可上网区提前下载；使用 `export OLLAMA_MODELS` 保存到持久化存储；准备备用模型下载链接 |
| **GPU 资源不足（70B 模型太大）** | 无法运行 70B | 中高 | 使用 GPTQ/AWQ 量化版本（如 `llama3.3:70b-instruct-q4_K_M`）；或降级到 30B-40B 模型 |
| **openai 库版本冲突** | 代码报错 | 中 | 指定 `openai==0.28.0` 版本；或使用 `requests` 直接调 REST API 绕过 SDK |
| **pydantic 版本冲突** | ollama-python 与项目冲突 | 中 | 不安装 `ollama-python`，直接用 `openai` SDK + Ollama 兼容接口 |
| **单次 run 超时 (>10min)** | 任务判定失败 | 中高 | 减少 LLM 调用次数（如反思机制最多调用 2 次）；避免过长的 Prompt；7B 模型比 70B 快很多 |
| **xvfb 虚拟显示器未启动** | 运行崩溃 | 低 | 记住命令: `xvfb-run python run_dialog.py --skip_display` |
| **在 GPU 区无法下载模型** | Phase 3 卡死 | 中 | 先在可上网区下载模型到持久化存储，再切换到 GPU 区加载 |
| **镜像打错名/选错版本** | 环境完全不可用 | 低 | 严格核对镜像名: `ubuntu2204-roco-exam:1.0`；提前用 `conda list` 检查关键包 |
| **实验结果不可复现（LLM 随机性）** | 评测分数波动 | 中 | 设置 `temperature=0.0`；多次运行取平均；固定 random seed |
| **小组分工不清晰** | 效率低下 | 中 | 尽早按任务或创新方向分工；使用 Git 分支协作 |

### B. 调参速查表

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| `temperature` | 0.0 | 越低越确定，减少随机性，提高可复现性 |
| `max_tokens` | 1024-2048 | 太小会截断输出，太大浪费时间和显存 |
| `top_p` | 0.9 | 通常不需要改，保持默认 |
| `num_runs` | 5-10 | 测试时用 1，正式评估用 5-10 |
| `num_ctx` | 4096 | Ollama 上下文长度，确保大于 Prompt 总 token 数 |

### C. 常见错误排查

| 错误现象 | 可能原因 | 解决方法 |
|---------|---------|---------|
| `import mujoco` 报错 | conda 环境未激活 | `conda activate roco` |
| Ollama 连接被拒绝 | Ollama 服务未启动 | `ollama serve` |
| OpenAI API 返回 404 | model 名称拼写错误 | `ollama list` 确认正确的模型名 |
| 输出动作解析失败 | LLM 输出格式不符合预期 | 加强 Prompt 中输出格式约束 + 添加格式纠错逻辑 |
| xvfb 未找到 | xvfb 未安装 | `sudo apt install xvfb` |
| 内存不足 | 70B 模型太大 | 使用量化版本或降级模型 |
| 任务超时 | LLM 调用次数过多 | 减少反思轮次、压缩 Prompt、使用更快的模型 |
