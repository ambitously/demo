# 70B大模型云服务器部署 —— 手把手从零教程

> 适用场景：具身智能竞赛，在云平台 GPU 实例上部署 70B 级大语言模型（如 llama3.3:70b）
> 前置基础：你只需要会用 API 调用大模型即可，本教程从零带你走完整个部署流程

---

## 前置知识

### 什么是 70B 模型？参数量意味着什么？

大语言模型（LLM）本质上是一个巨大的神经网络，"B"代表 **Billion（十亿）**，70B 就是指模型有 **700 亿个参数**。

打个比方：

| 模型规模 | 参数量 | 类比 | 能力表现 |
|---------|-------|------|---------|
| 7B | 70 亿 | 小学生读过几本书 | 能回答简单问题，复杂推理容易出错 |
| 70B | 700 亿 | 博士生读了整个图书馆 | 逻辑推理、代码生成、多轮对话显著更优 |

简单来说：**参数越多，"脑子越大"，理解力和推理力越强**。对于具身智能竞赛中的复杂任务规划、长程推理，70B 模型相比 7B 模型有质的飞跃。

### 为什么需要 GPU？显存怎么算？

大模型运行时，所有参数需要加载到显存（VRAM）中。显存需求大致计算如下：

```
模型大小 = 参数量 × 每个参数的字节数

例如 70B 模型：
  FP16（半精度）： 70 × 10^9 × 2 bytes ≈ 140 GB
  INT4（4-bit量化）：70 × 10^9 × 0.5 bytes ≈ 35 GB
```

实际运行中还需要额外显存用于上下文（KV Cache），所以推荐配置：

| 精度 | 最低显存 | 推荐显存 |
|------|---------|---------|
| FP16 | ~140 GB | 160+ GB（如 8×A100 80GB） |
| INT4 量化 | ~35 GB | 40-48 GB（如单张 A100 40GB / L40S） |

> **结论**：除非你有 8 张 A100，否者必须使用量化版本才能跑 70B。Ollama 默认下载的就是 4-bit 量化版本，一卷 A100（40GB/48GB）即可运行。

### 为什么要从 7B 开始测试，最后换 70B？

这是竞赛实战中的最佳策略：

1. **7B 模型下载快**（~4GB，几分钟下完），可以立刻验证整体流程是否跑通
2. **7B 推理快**，调试代码时不用等半天，迭代效率极高
3. **最后一天切 70B**：流程验证没问题后，下载 70B 模型（~40GB，需要较长时间），用正式数据跑最终结果

> ⚠️ 不要一上来就下 70B。一定先用 7B 把代码调通，确认整个 pipeline 没问题，再切大模型。

## 第三步：下载 70B 模型（关键步骤）

> ⚠️ **竞赛重要提示**：云平台的"可上网区"和"GPU区"是分开的。你需要在**可上网区**下载模型，但在**GPU区**运行。如果模型保存到默认路径，系统打包镜像时会把它打进镜像，浪费大量时间和空间。因此必须修改存储路径。

### 为什么必须修改 OLLAMA_MODELS 环境变量？

Ollama 默认把模型保存在 `~/.ollama/models/`，这个路径在系统盘上。问题在于：

1. **系统盘空间通常很小**（40-100GB），而 70B 模型约 40GB，可能直接撑爆
2. **竞赛平台的 GPU 区实例是临时创建的**，系统盘会被重置，每次重建都要重新下载 70B 模型（几十分钟到几小时，不现实）
3. **正确做法**：把模型保存到平台的**持久化存储区**（通常是挂载的数据盘 `/data` 或 `/mnt/storage`），GPU 区可以直接加载

### 设置 OLLAMA_MODELS

```bash
# 1. 先在存储区创建模型目录（示例路径，请替换为你的实际存储路径）
mkdir -p /data/ollama_models

# 2. 设置环境变量
export OLLAMA_MODELS="/data/ollama_models"

# 3. 重新启动 Ollama 使环境变量生效
sudo systemctl stop ollama
export OLLAMA_MODELS="/data/ollama_models"
ollama serve &

# 4. 验证路径设置
ollama list
# 模型将保存到 /data/ollama_models/ 而非默认的 ~/.ollama/models/
```

> 💡 **永久生效**：可以把 `export OLLAMA_MODELS="/data/ollama_models"` 写入 `~/.bashrc`：
> ```bash
> echo 'export OLLAMA_MODELS="/data/ollama_models"' >> ~/.bashrc
> source ~/.bashrc
> ```

```bash
ollama run llama3.3:70b "你好，请介绍你自己"
```

### Python 调用测试（OpenAI 兼容接口）

Ollama 提供了与 OpenAI API 完全兼容的接口，你可以直接用 `openai` 库调用：

```python
from openai import OpenAI

# 连接本地 Ollama 服务
client = OpenAI(
    base_url="http://localhost:11434/v1",  # Ollama 默认端口 11434
    api_key="ollama"  # Ollama 不需要真实 key，随便填
)

response = client.chat.completions.create(
    model="llama3.3:70b",
    messages=[
        {"role": "system", "content": "你是一个具身智能助手，帮助机器人完成任务规划。"},
        {"role": "user", "content": "桌子上有一个苹果，请规划抓取它的步骤。"}
    ],
    temperature=0.7,
    max_tokens=1024
)

print(response.choices[0].message.content)
```

### 如果推理太慢怎么办


| 量化等级 | 大小 | 速度 | 质量 |

| Q4_K_M | ~40GB | 中等 | 几乎没有损失 |
| Q3_K_M | ~30GB | 快 | 轻微损失 |
| Q2_K | ~24GB | 最快 | 明显损失，不推荐 |

## 第五步：在竞赛代码中对接

### 从 7B 切到 70B —— 只需要改一行

假设你的 7B 测试代码是这样的：

```python
# test.py —— 用 7B 调试的代码
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:11434/v1",
    api_key="ollama"
)

response = client.chat.completions.create(
    model="llama3.2:3b",  # <-- 7B 测试模型
    messages=[...]
)
```

切到 70B 只需要把 model 参数改了：

```python
    model="llama3.3:70b",  # <-- 就改这一行！
```

### 确认推理速度满足 10 分钟时间限制

竞赛通常有时间限制（例如每个任务不超过 10 分钟）。你需要估算：

```
70B 模型生成速度约 20 tokens/s

一次完整的任务规划 + 代码生成，假设需要 2000 tokens：
2000 tokens ÷ 20 tokens/s = 100 秒 ≈ 1.7 分钟

加上 prompt 处理时间、环境交互等，总计通常不会超过 5 分钟。
✅ 满足 10 分钟限制。
```

如果你需要生成大量内容（如写几千行代码），可以适当调整 `max_tokens` 限制：

```python
response = client.chat.completions.create(
    model="llama3.3:70b",
    messages=[...],
    max_tokens=2048,  # 控制单次最大生成量，避免超时
    timeout=300  # 5分钟超时保护
)
```

### 建议的开发流程

```
Day 1-3    → 用 7B 模型开发调试，快速迭代
Day 4-5    → 代码基本稳定后，切换到 70B 测试一下流程
Day 6      → 在可上网区下载 70B 模型到持久化存储
Day 7      → 用 70B 模型跑正式结果；如果出问题，至少还有 7B 保底
```

> 🔥 **最重要的建议**：永远保证你有一个能完整跑通的 7B baseline！这样即使 70B 部署出问题，你也不是零分提交。

## 常见问题排查

### 1. 下载中断怎么办？

Ollama 支持**断点续传**，直接重新运行下载命令即可：

```bash
# 如果下载中断，再次运行同样的命令即可从断点继续
ollama pull llama3.3:70b

# 输出中会显示已下载的进度继续增长
```

### 2. 显存不足（OOM）怎么办？

```bash
# 如果运行时报类似错误：
# "CUDA out of memory. Tried to allocate xxx MiB..."

# 解决方案一：使用更小量化版本
ollama pull llama3.3:70b-instruct-q3_K_M

# 解决方案二：降低上下文长度
ollama run llama3.3:70b
# 进入交互模式后：
/set parameter num_ctx 2048   # 默认 2048，可以进一步降低到 1024
/save my_70b_low_ctx

# 解决方案三：降级到中等模型
ollama pull llama3.3:32b      # 32B 只需要 ~18GB 显存
```

### 3. 推理太慢怎么办？

```bash
# 检查 GPU 是否真的在被使用
nvidia-smi
# 如果 GPU-Util 接近 0%，说明模型跑在 CPU 上，没用到 GPU

# 检查 ollama 是否正确使用 GPU
ollama run llama3.3:70b
# 在聊天界面输入：
/bye
# 看日志：cat ~/.ollama/logs/

# 如果 GPU 一直没被使用，可能是驱动问题：
nvidia-smi                 # 确认驱动正常
nvcc --version             # 确认 CUDA 可用
ollama --version           # 确认 Ollama 版本支持 GPU
```

> 💡 Ollama 会自动检测 GPU 并使用。如果是 Docker 容器，确保启动时加了 `--gpus all`。

### 4. 模型路径问题

```bash
# 检查模型保存在哪里
ollama list
# 然后检查环境变量
echo $OLLAMA_MODELS

# 如果为空，说明模型在默认路径 ~/.ollama/models/
ls -lh ~/.ollama/models/

# 如果设置了环境变量但 Ollama 还是读默认路径：
# 确保环境变量在启动 ollama serve 之前就设置好了
# 关了 Ollama 重开即可
pkill ollama
export OLLAMA_MODELS="/data/ollama_models"
ollama serve &
```

### 5. Ollama 服务无法启动

```bash
# 常见原因一：端口被占用
lsof -i :11434          # 查看谁占了 11434 端口
kill <PID>               # 杀掉占用的进程

# 常见原因二：权限问题
sudo chown -R $USER:$USER $OLLAMA_MODELS   # 修复模型目录权限

# 常见原因三：systemd 冲突
# 如果你之前用 sudo systemctl start ollama 启动过
# 又用 ollama serve & 启动，会冲突
sudo systemctl stop ollama  # 先停掉 systemd 版本
ollama serve &              # 再用命令行启动

# 查看日志排查
journalctl -u ollama -f    # 查看 systemd 日志
# 或
tail -f ~/.ollama/logs/     # 查看 Ollama 自身日志
```

### 6. 在 GPU 区加载不到模型

```bash
# 场景：在可上网区下载了模型，切到 GPU 区后 ollama list 显示为空

# 原因：GPU 区实例是新的，没有设置 OLLAMA_MODELS 环境变量
# 解决：
export OLLAMA_MODELS="/data/ollama_models"  # 指向你持久化存储的路径
ollama serve &
ollama list   # 应该能看到之前下载的模型了
```

### 7. Docker 容器中使用 Ollama

如果竞赛平台的镜像 `ubuntu2204-roco-exam:1.0` 是基于 Docker 的：

```bash
# 检查是否在 Docker 容器中
cat /proc/1/cgroup | grep docker

# 如果在 Docker 中，启动容器时需要映射模型存储
# 在宿主机（可上网区）：
docker run --gpus all \
  -v /data/ollama_models:/root/.ollama \
  -p 11434:11434 \
  ubuntu2204-roco-exam:1.0
```

## 附录：模型选择对比

### 什么是量化（Quantization）？

量化就像把高清电影压成 720P —— 文件小了很多，但内容基本看不出来差别。技术上讲，是把 16 位浮点数"压缩"成 4 位整数，模型大小变为原来的四分之一，显存需求也只要四分之一。

Ollama 的量化标签含义：
- **没有量化标签**（如 `llama3.3:70b`）：默认 Q4_K_M —— 推荐
- `instruct-q4_K_M`：微调版 + Q4 量化，K_M 是"中等质量"的量化方案
- `instruct-q4_0`：最基础的 4-bit 量化
- `instruct-q5_K_M`：5-bit 量化，质量更好但更大

### 主要 70B 级模型对比

| 特性 | llama3.3:70b | qwen2.5:72b | deepseek-r1:70b |
|------|-------------|------------|-----------------|
| 开发商 | Meta | 阿里（Qwen） | DeepSeek |
| 参数量 | 70B | 72B | 70B |
| 量化后大小 | ~40GB | ~41GB | ~40GB |
| 中文能力 | ★★★★ | ★★★★★ | ★★★★ |
| 英文能力 | ★★★★★ | ★★★★ | ★★★★ |
| 代码能力 | ★★★★★ | ★★★★ | ★★★★★ |
| 推理/思维链 | ★★★★ | ★★★★ | ★★★★★ |
| 社区生态 | ★★★★★ | ★★★★ | ★★★★ |
| 下载热度 | 最高 | 高 | 增长中 |

### 推荐选择

**首选：`llama3.3:70b`**（综合最强，生态最好，文档最多，出问题最容易搜到解决方案）

**如果任务偏中文** → `qwen2.5:72b`（阿里出品，中文训练数据更多）

**如果任务需要强推理** → `deepseek-r1:70b`（思维链能力突出，适合复杂多步规划，但速度可能稍慢）

**如果显存不够** → 退而求其次选择：
- `llama3.3:32b`（~18GB，大部分场景够用）
- `qwen2.5:32b`（~18GB，中文能力依旧出色）
