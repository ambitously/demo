# 多机器人协同操作：开源方案调研、资源汇总与10轮方案迭代

> 面向题目：**多机器人协同的具身操作 / RocoBench / MuJoCo / Ollama LLM 接入**  
> 生成日期：2026-05-16  
> 当前约束：本地暂时没有真实 GPU 卡和官方仿真环境，因此本文件做的是**资料调研 + 离线方案迭代 + 到服务器后的验证计划**。

---

## 0. 先给结论

这道题最值得借鉴的不是“训练模型”，而是下面几类开源方案的方法论：

1. **RoCo / RocoBench**：这是原题最接近的开源方案，核心是多机器人对话、子任务计划、环境反馈、重规划。
2. **SayCan**：不要让 LLM 随便生成动作，而是从“当前可执行动作列表”里选，避免幻觉动作。
3. **Inner Monologue / Reflexion**：执行失败后，把环境反馈写回 prompt，让 LLM 自我反思并重规划。
4. **LLM-Planner / ReAct / CoT**：先分解任务、再逐步执行，避免长程任务一次性规划崩掉。
5. **AutoGen / 多智能体辩论**：把机器人分成多个 Agent，让 Planner、Critic、Coordinator 分工协作。
6. **Ollama OpenAI Compatibility**：用 OpenAI 兼容接口最稳，避免 `ollama-python` 和旧项目依赖冲突。

**最终推荐方案**：

```text
Ollama 70B / 7B
  ↓
结构化 Prompt：OBSERVE → ANALYZE → ROLE_ASSIGN → PLAN → ACTION
  ↓
动作合法性过滤：只允许从 valid actions / skill space 中选
  ↓
多机器人角色分工：谁拿、谁扫、谁等、谁开门
  ↓
失败反思：失败后自动生成原因 + 修正策略
  ↓
日志评估：记录每次失败类型，迭代 Prompt
```

## 3. 资源方法总结：我们应该怎么吸收

### 3.1 RoCo → 我们的主方案骨架

RoCo 给我们的启发：

```text
机器人各自观察 → 多机器人对话 → 生成子任务计划 → 底层执行 → 环境反馈 → 重新对话修正
```

我们在竞赛里可以简化成：

```text
格式化 observation
  ↓
LLM 生成结构化计划
  ↓
解析 ACTIONS
  ↓
仿真执行
  ↓
如果失败，把失败信息加入下一轮 prompt
```

### 3.2 SayCan → 防止 LLM 胡编动作

LLM 最大问题是会输出看似合理但环境不接受的动作。因此要把 prompt 从：

```text
请你决定下一步动作
```

改成：

```text
以下是当前可选动作列表，请只从中选择一个或多个，不允许发明新动作。
```

这就是本题最实用的稳定性提升点。

### 3.3 Inner Monologue / Reflexion → 失败后不要直接结束

如果发生：碰撞、抓错、放错、超出范围、重复动作，应该让 LLM 看到：

```text
上一步动作失败：rob0: PICK red_cube
失败原因：red_cube 不在 rob0 可达范围内
请解释失败原因，并重新选择一个可执行动作。
```

这会显著提升复杂任务的鲁棒性。

### 3.4 AutoGen / Debate → 创新展示很好看，但实现要轻量

不要真的引入 AutoGen，容易依赖爆炸。我们只实现轻量版本：

```text
Planner：提出动作
Critic：检查是否冲突/非法/重复
Coordinator：修正为最终动作
```

这可以完全用三次 Ollama API 调用实现。

## 4. 基于开源方法的10轮方案迭代

下面是我根据上述开源方案，对当前方案进行的**10轮离线迭代**。每一轮都包含：借鉴来源、改进目标、可本地完成内容、到服务器后验证方式、预期收益。

---

### 第1轮：最小可运行 API 调用链

**借鉴来源**：RoCo baseline、Ollama OpenAI compatibility  
**目标**：先不要追求高分，只让 `dialog_prompter.py` / `plan_prompter.py` 能稳定调用本地模型并返回字符串。

#### 本地已准备
- `code/ollama_client.py`：OpenAI 兼容接口封装。
- `code/test_ollama.py`：连接测试脚本。
- `code/config.py`：模型名和 base_url 配置。

#### 核心代码形态

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:11434/v1",
    api_key="ollama"
)

response = client.chat.completions.create(
    model="qwen2.5:7b",
    messages=[{"role": "user", "content": prompt}],
    temperature=0.1,
)
text = response.choices[0].message.content
```

#### 到服务器后验证

```bash
conda activate roco
ollama serve
ollama pull qwen2.5:7b
python code/test_ollama.py --model qwen2.5:7b
```

#### 预期收益
- 端到端链路跑通。
- 避免 API 接口和依赖问题卡死。

---

### 第2轮：结构化输出约束

**借鉴来源**：Prompt Engineering Guide、Ollama JSON mode、Function Calling 思路  
**目标**：让 LLM 输出固定标签，便于解析。

#### 改进前

```text
机器人应该移动到蓝色方块附近，然后扫它。
```

这种自然语言不好解析。

#### 改进后

```text
<THINK>
分析当前状态...
</THINK>
<ACTIONS>
rob0: MOVE blue_cube
rob1: MOVE blue_cube
</ACTIONS>
```

#### 本地已准备
- `prompt_templates.py` 中的 `parse_llm_response()`。
- 6个任务都已写入 `<THINK>` + `<ACTIONS>` 格式要求。

#### 到服务器后验证
- 统计 LLM 输出中能否成功解析出动作。
- 指标：`parse_success_rate >= 95%`。

#### 预期收益
- 减少“输出看起来对但程序读不懂”的问题。

---

### 第3轮：JSON修复与格式自愈

**借鉴来源**：Toolformer / Structured Output / JSON repair  
**目标**：如果 LLM 输出格式不合规，不直接失败，而是二次修复。

#### 本地可加模块

```python
def repair_action_output(raw_output, client):
    prompt = f"""
下面的机器人动作输出格式不正确，请修复成严格格式。
不要解释，只输出 <ACTIONS> 标签。

原始输出：
{raw_output}
"""
    return client.chat([{"role": "user", "content": prompt}])
```

#### 到服务器后验证
- 记录修复前解析失败数。
- 记录修复后是否能解析。

#### 预期收益
- 中途崩溃减少。
- 新手最容易靠这个拿到稳定分。

---

### 第4轮：动作合法性过滤 / SayCan化

**借鉴来源**：SayCan  
**目标**：把 LLM 的自由生成改成“从可执行动作中选择”。

#### 核心思想

```text
LLM 不再创造动作，而是选择 action_id。
```

#### Prompt 模板

```text
当前任务：{task}
当前状态：{state}

你只能从以下动作中选择：
0. rob0: MOVE red_cube
1. rob1: WAIT
2. rob0: SWEEP red_cube

请输出：
{"action_id": 0}
```

#### 服务器上需要接入的点
- 查看原框架是否有 skill space / available actions。
- 如果没有，就根据 task prompt 中的动作空间手动构造候选。

#### 验证指标
- 非法动作率 `invalid_action_rate`。
- 目标：比第1轮下降 50% 以上。

#### 预期收益
- 这是最稳、最有工程价值的一轮。

---

### 第5轮：任务分解为子目标

**借鉴来源**：LLM-Planner、CoT、Hierarchical Planning  
**目标**：避免一次性规划完整任务导致长程错误。

#### 示例：Sweep Floor

```json
{
  "subgoals": [
    "移动两个机器人到 red_cube 两侧",
    "扫 red_cube 到簸箕",
    "将簸箕移动到垃圾桶",
    "倒入垃圾桶",
    "重复处理下一个 cube"
  ]
}
```

#### 本地可完成
- 给6个任务写固定子目标模板。
- 不依赖仿真环境。

#### 服务器后验证
- 每步只关注当前 subgoal。
- 记录每个 subgoal 是否完成。

#### 预期收益
- 对 Make Sandwich、Arrange Cabinet、Move Rope 特别有用。

---

### 第6轮：动态角色分配

**借鉴来源**：RoCo 多机器人对话、AutoGen role-based agents  
**目标**：每轮先决定每个机器人职责，减少抢同一物体或同时移动冲突。

#### 角色分配模板

```text
请先分配角色：
- 哪个机器人负责主要操作？
- 哪个机器人负责辅助？
- 哪个机器人需要等待？

输出：
<ROLES>
rob0: sweeper
rob1: dustpan_holder
</ROLES>
```

#### 本地可完成
- 在 `prompt_templates.py` 添加 `ROLE_ASSIGNMENT_PROMPT`。
- 每个任务写默认角色：
  - Sweep Floor：扫帚机器人 / 簸箕机器人
  - Arrange Cabinet：左门 / 右门 / 取杯子
  - Move Rope：左端 / 右端

#### 服务器后验证
- 观察是否减少两个机器人同时抓同一物体。
- 统计 WAIT 动作是否更合理。

#### 预期收益
- 协同逻辑更清晰，报告里很好展示。

---

### 第7轮：失败反思与重规划

**借鉴来源**：Inner Monologue、Reflexion  
**目标**：失败后不硬跑，主动分析失败原因并修正。

#### 反思 Prompt

```text
上一步失败：{failed_action}
失败原因：{error_message}
当前状态：{current_state}

请回答：
1. 为什么失败？
2. 下一步应该改成什么动作？

输出格式：
<REFLECTION>...</REFLECTION>
<ACTIONS>...</ACTIONS>
```

#### 本地已准备
- `prompt_templates.py` 中已有 `SELF_REFLECTION_PROMPT`。
- `dialog_handler_template.py` 里有 `record_failure()` 思路。

#### 服务器后验证
- 统计失败后恢复成功率 `recovery_rate`。

#### 预期收益
- 对复杂任务和低成功率任务帮助大。

---

### 第8轮：短期记忆与循环检测

**借鉴来源**：Voyager skill library、Reflexion episodic memory  
**目标**：避免 LLM 重复执行无意义动作。

#### 循环检测规则

```python
def is_loop(history, k=3):
    if len(history) < k:
        return False
    return len(set(map(str, history[-k:]))) == 1
```

#### 循环警告 Prompt

```text
警告：你已经连续多次执行相同或无效动作。
请不要重复历史动作，换一个策略。
```

#### 到服务器后验证
- 指标：重复动作率、连续失败次数。

#### 预期收益
- 减少超时失败。

### 第9轮：多候选投票 / Self-Consistency

**借鉴来源**：Self-Consistency、Multi-Agent Debate  
**目标**：同一状态生成多个候选动作，再由 Critic 选择最稳的。

#### 轻量实现

```python
candidates = []
for i in range(3):
    out = client.chat(prompt, temperature=0.5)
    candidates.append(parse(out))

best = critic_select(candidates, state, task)
```

#### Critic Prompt

```text
以下是3个候选动作，请选择最安全、最可能成功的一个。
重点检查：是否越界、是否碰撞、是否重复、是否符合当前子目标。
```

#### 注意
- 这会增加 LLM 调用次数，70B 模型可能变慢。
- 建议只在关键步骤或失败后启用。

#### 到服务器后验证
- 比较单次调用 vs 3候选投票的成功率和耗时。

#### 预期收益
- 提升关键动作稳定性。

### 第10轮：最终组合方案 V10

**借鉴来源**：RoCo + SayCan + Reflexion + Self-Consistency  
**目标**：把前9轮效果最稳的模块组合成最终可提交方案。

#### 最终架构

```text
读取 observation
  ↓
格式化状态文本
  ↓
任务分解 / 当前子目标判断
  ↓
角色分配
  ↓
生成候选动作列表 / 动作合法性过滤
  ↓
LLM选择动作
  ↓
格式解析与修复
  ↓
执行动作
  ↓
记录日志
  ↓
失败则反思重规划；成功则进入下一步
```

#### 最终 Prompt 骨架

```text
你是多机器人协同规划器。

必须遵守：
1. 只从可用动作中选择，不许发明动作。
2. 优先完成当前子目标。
3. 避免重复失败动作。
4. 如果有碰撞风险，让一个机器人 WAIT。
5. 输出必须严格符合 <THINK> 和 <ACTIONS> 格式。

任务：{task}
当前子目标：{subgoal}
机器人角色：{roles}
当前状态：{state}
最近历史：{history}
可用动作：{valid_actions}

请输出：
<THINK>...</THINK>
<ACTIONS>...</ACTIONS>
```

#### 最终报告可展示的亮点

1. **结构化 CoT**：提升动作可解释性。
2. **动作合法性过滤**：降低 LLM 幻觉。
3. **动态角色分配**：提升多机器人协同效率。
4. **失败反思机制**：提升鲁棒性。
5. **日志驱动迭代**：有工程闭环，而不是只调 prompt。

#### 服务器上最终验证命令

```bash
python run_dialog.py --task sweep_floor --num_runs 5 --skip_display
python run_dialog.py --task sort_cubes --num_runs 5 --skip_display
python evaluator.py
```
