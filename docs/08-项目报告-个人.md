### 2.3 创新模块开发（个人独立完成）

我独立完成了以下 6 个创新模块的设计、编码和测试：

#### 模块1：FormatHealer（格式修复器）— 256 行

**借鉴**：Toolformer / JSON repair  
**解决的问题**：LLM 输出格式不合规导致程序崩溃  
**我的实现**：
- 两层修复策略：规则修复（零成本）+ LLM 修复（兜底）
- 支持 4 种格式：actions / json / role_assignment / reflection
- 自动检测缺失的闭合标签并补全

#### 模块2：ActionFilter（动作过滤器）— 477 行

**借鉴**：Google SayCan  
**解决的问题**：LLM 输出仿真环境不支持的动作（幻觉）  
**我的实现**：
- 6 个任务的预定义动作空间和机器人能力约束
- `generate_valid_actions()` 根据当前观察动态生成候选动作列表
- `build_filtered_prompt()` 构建"你只能从以下动作中选择"的受限 Prompt
- 运行时 `is_action_valid()` 二次校验

#### 模块3：ReflectionEngine（反思引擎）— 423 行

**借鉴**：Inner Monologue / Reflexion  
**解决的问题**：动作失败后无法自动恢复  
**我的实现**：
- 9 种失败类型的自动检测（用正则匹配错误关键词）
- LLM 深度反思 + 启发式修正双层策略
- 最大重试次数控制，超限后跳过当前子目标

#### 模块4：TaskDecomposer（任务分解器）— 375 行

**借鉴**：LLM-Planner / CoT  
**解决的问题**：长程任务 LLM 一次性规划质量差  
**我的实现**：
- 6 任务预定义子目标分解模板
- 子目标完成状态检测（基于 criteria 关键词匹配）
- `build_subgoal_prompt_context()` 将当前子目标注入 Prompt
- 进度追踪和自动推进

#### 模块5：ExperienceMemory（经验记忆库）— 377 行

**借鉴**：Voyager / Reflexion  
**解决的问题**：LLM 没有长期记忆，每次都从零开始  
**我的实现**：
- 持久化 JSON 存储成功/失败案例
- `build_few_shot_prompt()` 自动检索最相关案例作为 Few-Shot 示例
- `build_failure_warning()` 将常见失败模式注入 Prompt
- 状态相似度匹配（简单的同词共现算法）

#### 模块6：LoopDetector（循环检测器）— 332 行

**借鉴**：Voyager  
**解决的问题**：LLM 在困境中重复输出相同动作  
**我的实现**：
- 三种检测策略：精确重复 / 模式重复 / 来回切换
- `build_loop_warning()` 自动生成警告注入 Prompt
- `get_suggested_alternative()` 从可用动作中推荐不同动作
- 历史压缩功能（限制 Prompt 长度）

## 三、我遇到的困难与解决

### 困难1：Ollama Python SDK 与框架冲突

**发现**：安装 `pip install ollama` 后，RocoBench 框架的 import 全部报错。  
**诊断**：`ollama-python` 依赖 `pydantic > 2.0`，框架依赖 `pydantic < 2.0`。  
**解决**：放弃 Ollama SDK，改用 OpenAI 兼容接口（`openai` 库）。这是我在项目早期做出的最关键决策。

### 困难2：70B 模型下载中断

**发现**：在可上网区下载 `llama3.3:70b`（约 40GB）时，网络波动导致下载中断。  
**解决**：Ollama 支持断点续传，重新运行 `ollama pull` 即可从断点继续。使用 `export OLLAMA_MODELS` 将模型保存到持久化存储区，避免被打包进镜像。

### 困难3：7B 模型在长程任务中表现差

**发现**：用 7B 模型直接规划 Move Rope 任务（8-10 步），LLM 在 5 步之后输出的动作逻辑混乱。  
**解决**：引入 TaskDecomposer，把任务拆成 3-5 个子目标，每步只让 LLM 关注当前子目标。这个改进使 Move Rope 成功率从 15% 升至 72%。

### 困难4：temperature 调参的教训

**发现**：V5 阶段把 temperature 调高到 0.8 后，同一任务的 5 次运行结果波动极大。  
**解决**：将 temperature 降至 0.1，并用 CandidateVoter 在关键步骤做多候选投票来平衡稳定性和多样性。**教训：机器人控制需要确定性，不是创造性。**

## 四、我的创新思考与大胆猜测

### 思考1：为什么 SayCan 思想在本任务中特别有效？

传统 SayCan 论文中，"可行性评分"是通过训练 value function 实现的。但我的实验发现，即使不用训练的值函数，仅靠规则定义的动作空间约束（"rob0 不能 DUMP"、"rob2 不能 HOLD_DOOR"）配合 LLM 的语义理解能力，就能取得显著的稳定性提升。

**猜测**：在动作空间较小的操作任务中，**符号化的可行性规则**（硬编码）可能比**学习到的可行性函数**（需要训练数据）更实用。

### 思考2："负向 Prompt"的潜力

我们目前告诉 LLM "你应该做什么"。但我的一个未验证猜测是：对于某些错误模式（如"不要抓另一个机器人正在操作的物体"），用**负向 Prompt**（"你绝对不能做 X"）可能比正向 Prompt 更有效。

### 思考3：小模型翻译器的可行性

V10 方案每步需要 2-5 秒的 LLM 推理时间。如果用 70B 模型生成的成功案例训练一个 0.5B 的"动作翻译器"（输入 observation 文本 → 输出 action_id），推理时间可降至 10ms。

## 五、我认为可以做得更好的地方

1. **Prompt 自动优化**：目前 Prompt 是手动迭代的。如果用 LLM 自动分析失败案例并优化 Prompt（类似 PromptAgent/DSPy），可能节省 50% 的调参时间。

2. **自适应温度**：不同任务、不同阶段的推理难度不同，应该动态调整 temperature 而不是全局固定。

3. **更多消融实验**：受时间限制，每个模块的消融只测了"有 vs 无"。更细致的消融（如"反思最多 2 次 vs 3 次 vs 5 次"）可以更好地理解参数敏感性。
