## 一、领域经典问题与有价值的研究方向

### 1.1 十大经典未决问题

| # | 问题 | 一句话描述 | 与本赛题关系 |
|:---:|------|------|:---:|
| 1 | **语言→物理世界的接地（Grounding）** | LLM懂语言但不懂物理——不知道物体在哪、能不能够到、会不会碰 | ⭐⭐⭐⭐⭐ 核心问题 |
| 2 | **长程规划与错误累积** | 10步任务中第3步的小误差到第8步会放大到完全失败 | ⭐⭐⭐⭐⭐ 直接影响成功率 |
| 3 | **Sim-to-Real 迁移** | 仿真里跑得好的策略到真机器人上往往失败 | ⭐⭐ 本赛题仅在仿真 |
| 4 | **跨场景泛化** | 在厨房训练的策略到客厅还能用吗？ | ⭐⭐⭐ CALVIN ABC→D就是测这个 |
| 5 | **多机器人协同** | 谁来干？什么时候等？信息怎么共享？冲突怎么解决？ | ⭐⭐⭐⭐⭐ 本赛题主题 |
| 6 | **LLM幻觉与安全性** | LLM可能输出"把杯子扔到天花板"这种危险动作 | ⭐⭐⭐⭐ 动作过滤就是解决这个 |
| 7 | **高层规划 vs 底层控制 gap** | LLM说"抓起杯子"但机器人需要知道关节角度 | ⭐⭐⭐ RocoBench已封装 |
| 8 | **评估基准缺失** | 没有公认的长程操纵评测标准 | ⭐⭐ CALVIN/RocoBench是探索 |
| 9 | **实时性约束** | LLM推理需要2-5秒，机器人控制需要毫秒级响应 | ⭐⭐⭐ 10分钟限制 |
| 10 | **数据稀缺** | 机器人交互数据比文本/图像数据少几个数量级 | ⭐ 本赛题不训练模型 |

### 1.2 五大最有价值的研究方向

#### 方向 A：LLM + 可行性过滤（SayCan路线）

**核心问题**：LLM会输出"语义合理但物理不可行"的动作。  
**解决方案**：先用LLM生成候选动作，再用一个"可行性判断器"过滤。  
**代表工作**：SayCan（Google, 2022），用预训练的价值函数评估动作可执行性。  
**为什么有价值**：这条路把LLM的"知识"和机器人的"能力"解耦了，是最实用的范式。  
**本赛题中的应用**：`ActionFilter` 模块。

**你可以深入的方向**：
- 如何在不训练价值函数的情况下实现可行性判断？（我们用规则实现了一个版本）
- 可行性判断能否用另一个更小的LLM来做？
- 如果LLM反复选中"可行但愚蠢"的动作怎么办？

---

#### 方向 B：闭环语言反馈（Inner Monologue路线）

**核心问题**：LLM一次性输出完整计划，中间出错了不知道。  
**解决方案**：每执行一步就把环境反馈（成功/失败/场景描述）追加到Prompt，让LLM"边看边调"。  
**代表工作**：Inner Monologue（Google, 2022），Reflexion（2023）。  
**为什么有价值**：这是从"开环规划"到"闭环控制"的质变。  
**本赛题中的应用**：`ReflectionEngine` 模块。

**你可以深入的方向**：
- 反馈应该详细到什么程度？（"抓取失败" vs "夹爪在x=0.5,y=0.3闭合但物体滑落"）
- 什么时候应该主动"提问"而不是被动接受反馈？
- 连续失败几次应该放弃当前子目标？

---

#### 方向 C：多智能体对话与辩论

**核心问题**：多机器人之间如何通信才能高效协作？  
**解决方案**：每个机器人有自己的LLM"大脑"，通过自然语言对话协商。  
**代表工作**：RoCo（Columbia, 2023），Multi-Agent Debate（2023），AutoGen（Microsoft）。  
**为什么有价值**：这是多机器人协作最自然的范式——就像人类团队开会讨论。  
**本赛题中的应用**：`dialog_prompter.py` 的 dialog 模式。

**你可以深入的方向**：
- 辩论几轮最优？（1轮不够，5轮太慢）
- 辩论中出现了意见分歧怎么仲裁？
- 如果有机器人"固执己见"怎么办？

---

#### 方向 D：代码作为策略（Code as Policies）

**核心问题**：自然语言描述动作有歧义，代码更精确。  
**解决方案**：让LLM直接写Python代码来控制机器人（调用API）。  
**代表工作**：Code as Policies（Google, 2022），Voyager（NVIDIA, 2023）。  
**为什么有价值**：代码可以处理循环、条件、函数调用，比文本动作序列表达力强得多。  
**本赛题中的应用**：可探索——让LLM生成Python代码而非动作文本序列。

---

#### 方向 E：世界模型（World Model）

**核心问题**：LLM规划时无法"想象"动作的后果。  
**解决方案**：训练一个"世界模型"预测动作执行后的状态，让LLM在"想象空间"里试错。  
**代表工作**：Dreamer系列，DayDreamer，UniSim。  
**为什么有价值**：这是实现"提前规划"的关键——就像人类下棋时在大脑里推演。  
**本赛题中的应用**：难度较大，但可作为"未来工作"讨论。

## 二、必须了解的经典架构

### 2.1 架构全景图

```
                         ┌─────────────────────────────┐
                         │    LLM + 机器人 架构全景      │
                         └─────────────────────────────┘
                                       │
          ┌────────────────────────────┼────────────────────────────┐
          │                            │                            │
          ▼                            ▼                            ▼
┌──────────────────┐    ┌─────────────────────┐    ┌──────────────────┐
│  规划层架构       │    │   策略层架构          │    │  端到端架构       │
│ LLM决策+执行分离  │    │  LLM直接输出动作      │    │  视觉→语言→动作   │
└──────────────────┘    └─────────────────────┘    └──────────────────┘
          │                            │                            │
    ┌─────┴─────┐              ┌──────┴──────┐             ┌───────┴───────┐
    │           │              │             │             │               │
    ▼           ▼              ▼             ▼             ▼               ▼
 SayCan    LLM-Planner     Code-as-      ReAct         RT-2           PaLM-E
                         Policies                    (VLA)       (Embodied VLM)
```

### 2.2 详细架构说明

#### 架构 1：SayCan — 规划+可行性过滤

```
用户指令："我打翻了可乐，帮我清理"
        │
        ▼
┌──────────────────┐
│  LLM (PaLM/GPT)  │ ← 评估每个技能对任务的"有用程度"
│  语义规划器       │
└──────┬───────────┘
       │ 有用度分数
       ▼
┌──────────────────┐
│  Value Function  │ ← 评估每个技能在当前状态下的"可行性"
│  可行性评估器     │
└──────┬───────────┘
       │ 可行性分数
       ▼
┌──────────────────┐
│  综合选择         │ ← 选"有用 ∩ 可行"最高的技能
│  argmax(有用×可行) │
└──────┬───────────┘
       │
       ▼
  机器人执行 → 更新状态 → 循环
```

**关键思想**：LLM 不管物理，只管语义；物理可行性由专门的模块负责。  
**为什么你必须理解**：这是本赛题 ActionFilter 模块的直接思想来源。

---

#### 架构 2：RT-2 — Vision-Language-Action（VLA）模型

```
摄像头图像 + 语言指令
        │
        ▼
┌──────────────────────┐
│  Vision Transformer  │ ← 视觉编码器
│  (ViT / PaLI-X)      │
└──────┬───────────────┘
       │ 视觉token
       ▼
┌──────────────────────┐
│  Language Model      │ ← 大语言模型（处理文本+视觉token）
│  (PaLM-E / T5)       │
└──────┬───────────────┘
       │ 动作token
       ▼
┌──────────────────────┐
│  Action Decoder      │ ← 把token翻译成机器人关节角度
└──────────────────────┘
```

**关键思想**：把机器人动作也编码成"语言token"，让LLM直接输出动作。  
**为什么你必须理解**：这是"端到端"路线的代表作，与"分层规划"路线形成对比。

---

#### 架构 3：Inner Monologue — 闭环反馈

```
┌─────────────────────────────────────────────────────────┐
│                    闭环控制循环                           │
│                                                         │
│   1. LLM 规划 ──→ 2. 机器人执行 ──→ 3. 观察结果          │
│       ↑                                    │             │
│       │                                    ↓             │
│       └──────── 5. 更新Prompt ←──── 4. 文本化反馈        │
│                                                         │
│   反馈类型：                                            │
│   - 场景描述："桌子上有一个红色方块和一个蓝色方块"        │
│   - 成功检测："抓取成功" / "抓取失败"                    │
│   - 碰撞信息："rob0和rob1在(0.5,0.5)发生碰撞"            │
│   - 人类反馈："别碰那个，先清理这个"                     │
└─────────────────────────────────────────────────────────┘
```

**关键思想**：不是"规划-执行"一次性完成，而是持续对话。  
**为什么你必须理解**：这是 ReflectionEngine 的思想来源。

---

#### 架构 4：VoxPoser — LLM + 3D价值地图

```
语言指令 + RGB-D图像
        │
        ▼
┌──────────────────┐
│  LLM (GPT-4)     │ ← 生成3D空间约束代码
│  生成Python代码   │
└──────┬───────────┘
       │ 代码调用VLM
       ▼
┌──────────────────┐
│  VLM             │ ← 识别物体→生成3D价值地图
│  物体识别+定位    │
└──────┬───────────┘
       │ 3D地图
       ▼
┌──────────────────┐
│  Motion Planner  │ ← 在价值地图上规划无碰路径
│  运动规划器       │
└──────┬───────────┘
       │
       ▼
  机器人执行轨迹
```

**关键思想**：LLM不直接规划动作，而是生成"约束"和"目标"让传统运动规划器去解。  
**为什么你必须理解**：这是一种优雅的分工——LLM做它擅长的（语义理解），规划器做它擅长的（运动学）。

---

### 2.3 你应该记住的架构口诀

## 三、初学者常见问题与解决方案

### 3.1 Ollama 相关

#### Q1：`ollama serve` 启动后在后台就停不下来？

**错误**：关了终端窗口 Ollama 就停了。  
**解决**：
```bash
# Linux 后台运行
nohup ollama serve > ollama.log 2>&1 &

# 或使用 systemd（推荐）
sudo systemctl enable ollama
sudo systemctl start ollama

# Windows 上 Ollama 安装后自动作为 Windows 服务运行
```

---

#### Q2：下载 70B 模型到一半断了怎么办？

**答**：Ollama **支持断点续传**。重新运行 `ollama pull llama3.3:70b` 即可从断点继续。

---

#### Q3：Ollama 模型下载完存在哪了？

**答**：
- Linux: `~/.ollama/models/`
- macOS: `~/.ollama/models/`
- Windows: `C:\Users\<用户名>\.ollama\models\`

**竞赛关键操作**：修改路径到持久化存储
```bash
export OLLAMA_MODELS="/your/persistent/storage/ollama_models"
```

---

#### Q4：`ollama-python` 库装了之后项目报错？

**诊断**：`ollama-python` 依赖 `pydantic >= 2.0`，很多旧项目依赖 `pydantic < 2.0`。  
**解决**：**不要用 `ollama-python`**，用 Ollama 的 OpenAI 兼容接口：
```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:11434/v1",
    api_key="ollama"
)
```

---

#### Q5：7B 模型能跑但 70B 报 OOM？

**诊断**：显存不够。  
**解决**：
1. 确认 GPU 有足够显存（70B 量化版需要 ~35-40GB）
2. 用 `ollama pull llama3.3:70b-instruct-q4_K_M` 指定更低量化版本
3. 如果只有 24GB 显存，用 32B 模型替代：`ollama pull qwen2.5:32b`

---

### 3.2 Python 环境相关

#### Q6：Conda 环境报 "command not found: conda"

**诊断**：Conda 没装或没在 PATH 里。  
**解决**：
```bash
# 检查是否已安装
which conda

# 如果没有，安装 Miniconda（最小化版本）
# Linux:
wget https://repo.anaconda.com/miniconda/Miniconda3-latest-Linux-x86_64.sh
bash Miniconda3-latest-Linux-x86_64.sh

# 安装后重启终端或 source ~/.bashrc
```

---

#### Q7：`pip install` 报依赖冲突

**常见场景**：`ERROR: pip's dependency resolver...`  
**解决（从易到难）**：
1. `pip install --upgrade pip`
2. 不 pin 死版本：`pip install package` 而不是 `pip install package==1.2.3`
3. 创建新环境：`conda create -n fresh python=3.8 && conda activate fresh`
4. 用 `pip install --no-deps package` 跳过依赖检查（危险，仅限最后手段）

---

#### Q8：import mujoco 报错

**常见原因**：
- 装错了包：应该 `pip install mujoco`（新版），不是 `pip install mujoco-py`（旧版已废弃）
- 缺少渲染库：`sudo apt-get install libosmesa6-dev libgl1-mesa-glx libglfw3`

---

### 3.3 Prompt Engineering 相关

#### Q9：LLM 的输出总是不符合我的格式要求

**最常见原因**（按频率排序）：
1. **System Prompt 写得太软**："请尽量用JSON格式输出" → 改成 "你**必须**只输出JSON，如有其他文字将导致系统崩溃"
2. **没有给正面示例**：在 Promot 中放一个正确的输出示例
3. **格式约束藏在长文本里**：把格式要求单独放在 Prompt 最后一行
4. **不同模型对格式遵守能力不同**：7B 模型格式遵守能力弱于 70B

---

#### Q10：换了模型后 Prompt 效果变差

**原因**：不同模型对 Prompt 的"口味"不同。  
**经验**：
- Qwen 系列对中文 Prompt 响应更好
- Llama 系列对英文 Prompt 响应更好
- 同样的 Few-Shot 示例，Qwen 需要 2-3 个，Llama 可能需要 4-5 个
- temperature=0.1 在不同模型上表现差异很大

**解决**：维护两套 Prompt 模板，根据模型名称自动选择。

---

### 3.4 竞赛特定

#### Q11：run_dialog.py 报 import 错误

**步骤排查**：
```bash
# 1. 确认在正确的 conda 环境
conda activate roco

# 2. 确认安装了依赖
pip install -r requirements.txt

# 3. 确认 Python 版本
python --version  # 应该是 3.8

# 4. 测试基础 import
python -c "import mujoco; print('mujoco OK')"
python -c "import dm_control; print('dm_control OK')"
```

---

#### Q12：xvfb 是什么？怎么用？

**答**：`xvfb` 是"虚拟显示器"——在没有物理屏幕的服务器上模拟一个显示器，让 MuJoCo 可以渲染。

```bash
# 安装
sudo apt-get install xvfb

# 使用
xvfb-run python run_dialog.py --task sweep_floor --skip_display
```

## 四、网上常见的困难与社区经验

### 4.1 GitHub Issues 高频问题

| 问题类型 | 出现频率 | 代表性描述 |
|----------|:---:|------|
| "Ollama JSON mode returns empty" | 🟡 常见 | Ollama issue #1135: 某些prompt下JSON输出卡死 |
| "Model not found" | 🔴 最常见 | 模型名拼写错误或未下载 |
| "CUDA out of memory" | 🔴 最常见 | 70B需要40GB显存 |
| "MuJoCo rendering black screen" | 🟡 常见 | 服务器无GPU/显示器 |
| "Conda environment broken after pip install" | 🟡 常见 | pip和conda混用导致 |
| "Python 3.8 vs 3.10 incompatibility" | 🟢 偶发 | 新库不支持Python 3.8 |

### 4.2 社区总结的经验教训

#### 经验 1：永远先用最小模型跑通流程

> "先用 0.5B 模型测试代码是否正确，再用 7B 模型测试 Prompt 是否有效，最后用 70B 模型跑正式结果。"  
> — 社区共识

#### 经验 2：Prompt 的"硬度"和"软度"

> "如果 LLM 输出格式不对，99% 的原因是你的格式约束不够硬。不要说'请尽量'，要说'必须、只能、绝对不要'。"  
> — r/LocalLLaMA 社区

#### 经验 3：不要相信 LLM 的"我会 JSON"

> "让 LLM 输出 JSON 就像让猫握手——有时候会，但不能指望。"  
> — 某 Hacker News 评论

#### 经验 4：机器人控制场景 temperature≈0

> "在需要确定性输出的场景（机器人控制、代码生成），temperature 应该尽可能低（0-0.2）。高温度适合写诗，不适合控制机械臂。"  
> — OpenAI Cookbook

#### 经验 5：模块化 > 单一大 Prompt

> "不要试图写一个'万能 Prompt'让它处理所有情况。把问题拆成小模块，每个模块解决一个特定问题。"  
> — LangChain 最佳实践

### 4.3 社区推荐的"避坑清单"

1. **不要在一个环境里同时用 conda 和 pip install** — 用 conda 装包，只在 conda 找不到时用 pip
2. **不要把模型下载到系统盘** — 70B 模型 40GB，系统盘满了机器会崩
3. **不要在 Prompt 里写"不要做X"** — LLM 对否定句的理解不稳定，改成"只能做Y"
4. **不要信任 LLM 的数学计算** — 如果需要距离计算，用 Python 算完再注入 Prompt
5. **不要把 API key 写在代码里** — 用环境变量或 config 文件
6. **不要在生产环境用 temperature > 0.3** — 除非你想要随机行为
7. **不要跳过 logger** — 没有日志的调试就像蒙着眼睛找东西
8. **不要假设"上次成功了这次也会成功"** — LLM 的输出有随机性

## 五、必不可少的补充知识

### 5.1 你必须理解的基础概念

#### 概念 1：LLM 的 Token 是什么？

> **Token ≠ 字**。一个 token 可能是一个字、一个词、或一个词的一部分。
> - 中文：大约 1.5-2 个 token = 1 个汉字
> - 英文：大约 1.3 个 token = 1 个单词
> - `qwen2.5:7b` 的上下文窗口是 32768 tokens（约 20000 汉字）
> - 你的 Prompt + LLM 回复不能超过这个限制

#### 概念 2：Temperature 到底控制什么？

> Temperature 控制 LLM 输出的**随机性**：
> - `0.0`：确定性输出（同一个 Prompt 每次都一样）
> - `0.3`：轻微随机性
> - `0.7`：明显随机性
> - `1.0`：高度随机（适合创意写作）
>
> **机器人控制用 `0.0-0.2`**，因为你需要稳定的、可预测的行为。

#### 概念 3：Prompt 的三个层次

```
System Prompt（系统提示词）
    ↓ 设定角色和全局规则
    "你是一个机器人规划专家..."

User Prompt（用户提示词）
    ↓ 描述当前任务和状态
    "当前任务是清扫地板。桌上有一个红色方块..."

Assistant（LLM回复）
    ↓ LLM的输出
    "<THINK>...</THINK><ACTIONS>...</ACTIONS>"
```

#### 概念 4：MuJoCo 是什么？为什么用它？

> MuJoCo（Multi-Joint dynamics with Contact）是 Google DeepMind 的开源物理引擎，专门用于机器人仿真。
> - 速度快（比 Gazebo 快 10-100 倍）
> - 接触力学准确
> - Python 原生绑定
> - 竞赛选择它是因为它免费、开源、快速

#### 概念 5：RocoBench 的通信模式

```
dialog 模式：
  每个机器人独立调用 LLM → 各自输出动作 → 可能有冲突 → 需要协调

plan 模式：
  一个 LLM 调用输出所有机器人的动作 → 天然协调 → 但对 LLM 要求更高
```

### 5.2 你在 Report 里可以引用的关键概念

| 概念 | 一句话解释 | 在哪引用 |
|------|------|------|
| **Affordance** | 环境给智能体提供的"可操作性"（这个门能开吗？这个杯子能抓吗？） | 动作过滤 |
| **Grounding** | 把抽象符号（"红色方块"）连接到物理实体（桌上的那个具体物体） | 问题分析 |
| **Chain-of-Thought** | 让LLM在输出答案前先输出推理过程 | Prompt设计 |
| **Few-Shot** | 在Prompt中给几个示例，让LLM模仿 | 经验记忆 |
| **Hallucination** | LLM输出看起来合理但实际错误的内容 | 动作过滤动机 |
| **Closed-loop** | 执行→观察→调整→再执行，而非一次性规划 | 反思引擎 |
| **Sim-to-Real Gap** | 仿真和真实世界之间的差距 | 局限讨论 |
| **Embodiment** | 智能体的"身体"（机械臂、轮子等）决定了它能做什么 | 背景介绍 |

### 5.4 5 个你必须知道的技术术语

1. **RRT (Rapidly-exploring Random Tree)**：机器人路径规划的最经典算法。RocoBench 底层用这个做避碰路径规划。
2. **IK (Inverse Kinematics)**：给定末端位置，反算关节角度。你不需要手写，但要知道存在这个东西。
3. **Gazebo**：另一个机器人仿真器，比 MuJoCo 慢但更"真实"。了解后能理解为什么竞赛选了 MuJoCo。
4. **ROS (Robot Operating System)**：机器人开发的"操作系统"。本题不需要，但真机上必用。
5. **URDF/MJCF**：机器人模型描述格式。MuJoCo 用 MJCF，ROS 用 URDF。
