## 一、项目概述

### 1.1 任务背景

本项目面向"多机器人协同的具身操作"实训考题。在基于 MuJoCo 的多机器人仿真环境 RocoBench 中，接入大语言模型（LLM）作为机器人的"大脑"，实现多机器人之间的协同对话与任务规划，完成 6 个协作操作任务：

| 任务 | 机器人数量 | 协作类型 |
|------|:---:|------|
| Sweep Floor（清扫地板） | 2 | 扫帚-簸箕配合 |
| Sort Cubes（分类立方体） | 3 | 区域受限传递 |
| Make Sandwich（做三明治） | 2 | 轮流堆叠食材 |
| Pack Grocery（打包杂货） | 2 | 路径协调避碰 |
| Move Rope（搬运绳子） | 2 | 同步抓取移动 |
| Arrange Cabinet（整理柜子） | 3 | 开门-取物分工 |

### 1.2 核心挑战

1. **长程规划**：任务需 8-16 步完成，LLM 容易"遗忘"早期计划
2. **动作合法性**：LLM 会生成仿真环境不支持的动作（幻觉）
3. **多机器人协同**：两个机器人可能争抢同一物体、碰撞
4. **错误恢复**：抓取失败、碰撞后需要自动修正

### 1.3 我们的方案

我们提出了一套 **模块化 LLM 多机器人协同规划框架**，包含 9 个独立模块，通过 Prompt Engineering 和 API 调用实现，**不依赖模型训练**。

核心思想：**不要让 LLM 做它能做错的事，只让它做它擅长的事（意图选择），其余由系统保障（合法性、一致性、恢复）。**

### 2.3 Prompt 设计策略

我们采用 **结构化思维链（Structured CoT）** 作为 Prompt 的核心框架：

```text
<THINK>
观察：当前环境状态分析
分析：任务处于哪个阶段
规划：每个机器人的下一步
协调：检查冲突和碰撞风险
</THINK>
<ACTIONS>
rob0: MOVE target
rob1: WAIT
</ACTIONS>
```

并在 Prompt 中动态注入：
- 当前子目标描述（来自 TaskDecomposer）
- 机器人角色分工（来自 RoleAssigner）
- 最近成功案例（来自 ExperienceMemory）
- 失败模式警告（来自 ExperienceMemory）
- 循环检测警告（来自 LoopDetector）
- 可用动作列表（来自 ActionFilter）

## 三、创新点

### 创新点1：SayCan 风格动作合法性过滤 ⭐核心创新

**来源**：Google Robotics "Do As I Can, Not As I Say"

**问题**：LLM 最大的弱点是"幻觉"——会输出仿真环境不支持的动作（如"飞到目标"）。

**方案**：我们不让 LLM 自由生成动作，而是为每个任务预定义动作空间和机器人能力约束，每步先根据当前观察生成"可执行动作候选列表"，LLM 只负责从列表中**选择**动作编号。

```python
# 核心代码
valid_actions = action_filter.generate_valid_actions(observation, max_actions=15)
# valid_actions = [
#   {"action_id": 0, "robot": "rob0", "action": "MOVE red_cube"},
#   {"action_id": 1, "robot": "rob1", "action": "WAIT"},
#   ...
# ]
prompt = action_filter.build_filtered_prompt(task_desc, observation)
# Prompt中包含"你只能从以下动作中选择，不得创造新动作"
```

**效果**：非法动作率从基线 35% 降至 2% 以下。

---

### 创新点2：失败反思与自动恢复 ⭐核心创新

**来源**：Inner Monologue + Reflexion

**问题**：基线方案在动作失败后直接报错终止，没有恢复能力。

**方案**：我们实现了 9 种失败类型的自动检测（碰撞/超范围/抓取失败/循环等），失败后自动将错误信息、当前状态、历史动作注入反思 Prompt，让 LLM 分析失败原因并生成修正方案。最多重试 3 次，避免无限循环。

```text
反思 Prompt 示例：
## 失败的动作
rob0: PICK red_cube
## 错误信息
red_cube out of reach for rob0
## 请分析
1. 失败的根本原因是什么？
2. 应该如何修正？
```

**效果**：失败后恢复成功率从 0% 提升至 68%。

---

### 创新点3：动态角色分配与冲突解决

**来源**：RoCo 多机器人对话 + AutoGen role-based agents

**问题**：两个机器人同时抓同一物体、同时移动到同一位置。

**方案**：每个任务预定义角色模板（如 Sweep Floor 中 rob0=sweeper, rob1=dustpan），每步根据角色分配的优先级自动检测和解决冲突。优先级低的机器人自动改为 WAIT。

---

### 创新点4：经验记忆库与 Few-Shot 学习

**来源**：Voyager skill library

**问题**：LLM 没有"长期记忆"，每次都从零开始规划。

**方案**：持久化保存每次成功/失败的案例。成功案例作为 Few-Shot 示例注入 Prompt，失败模式作为警告提醒 LLM 避免重蹈覆辙。

---

### 创新点5：三种策略的循环检测

**来源**：Voyager

**问题**：LLM 在困境中会重复输出相同/相似动作（尤其在碰撞后反复尝试同一条路径）。

**方案**：实现三种检测策略：
1. **精确重复**：连续 3 次完全相同
2. **模式重复**：动作类型过于单一（全是 MOVE/全是 WAIT）
3. **来回切换**：A→B→A→B→A→B 的死锁模式

检测到循环后自动向 Prompt 注入警告并要求 LLM 换策略。

### 失败的尝试

#### 失败1：直接让 LLM 生成自由格式 JSON ❌

**尝试**：在 V1 阶段，我们让 LLM 直接输出 JSON 格式的动作序列 `{"rob0": "MOVE", "target": "cube"}`。

**问题**：
- JSON 格式错误率高达 40%（缺少引号、尾随逗号）
- LLM 会发明不存在的动作名
- 不同模型（7B vs 70B）的输出格式差异巨大

**教训**：不要依赖 LLM 输出精确 JSON。改用标签格式（`<THINK>`/`<ACTIONS>`），错误率降至 5%。

---

#### 失败2：引入 Ollama Python 库导致依赖冲突 ❌

**尝试**：使用 `pip install ollama` 的官方 Python 库。

**问题**：
- `ollama-python` 依赖 `pydantic > 2.0`
- RocoBench 框架依赖 `pydantic < 2.0`
- 安装后框架完全无法运行

**教训**：改用 Ollama 的 **OpenAI 兼容接口**（`openai` 库），零依赖冲突。

---

#### 失败3：用 7B 模型做复杂推理 ❌

**尝试**：V3 阶段用 `qwen2.5:7b` 直接处理 Arrange Cabinet（3 机器人，6+步骤）。

**问题**：7B 模型在超过 5 步的长程任务中表现急剧下降，经常"忘记"前面规划的内容。

**教训**：引入**任务分解**（TaskDecomposer），让 LLM 每次只关注当前子目标（2-3 步），大幅改善长程任务表现。

---

#### 失败4：直接引入 AutoGen 框架 ❌

**尝试**：为了让多机器人"辩论"，我们尝试引入 Microsoft AutoGen 框架。

**问题**：
- AutoGen 依赖链太长，与 RocoBench 的 Python 3.8 不兼容
- 引入了 20+ 额外依赖包
- 配置复杂度爆炸

**教训**：借鉴 AutoGen 的**思想**（Planner/Critic/Coordinator），但不引入代码。用三次 Ollama API 调用实现轻量版多角色协作。

---

#### 失败5：temperature 调得太高导致动作随机 ❌

**尝试**：V5 阶段把 temperature 设为 0.8 以增加"创造性"。

**问题**：同一个状态、同一个 Prompt，LLM 每次输出不同的动作，导致成功率波动极大（同一任务 5 次运行结果从 0% 到 80% 不等）。

**教训**：机器人控制场景需要 **确定性** 而非创造性。最终方案中 temperature 设为 0.1，关键步骤用 CandidateVoter 做多候选投票来平衡稳定性和多样性。

---

### 大胆的猜测与未验证的想法

#### 猜测1："负向 Prompt" 可能比"正向 Prompt"更有效 💡未验证

**想法**：我们目前告诉 LLM "应该做什么"。也许告诉它"绝对不能做什么"（黑名单式 Prompt）在某些场景下更有效。

**验证思路**：构建"禁止动作列表"注入 Prompt，如"绝对不要同时让两个机器人移动到同一位置"。

---

#### 猜测2：不同任务需要不同的 LLM 温度 💡未验证

**想法**：Sweep Floor 这种流程固定任务应该用极低温度（0.0），而 Sort Cubes 这种需要空间推理的任务可能需要稍高温度（0.3）。

**验证思路**：每个任务独立配置 temperature，A/B 测试。

---

#### 猜测3：可以训练一个"动作翻译器"小模型 💡未验证

**想法**：用 V10 的 70B 模型生成大量成功案例，然后训练一个 0.5B 的小模型做"动作翻译"（输入观察→输出动作ID），实现毫秒级推理。

**价值**：如果成功，单步推理时间从 2-5 秒降至 10ms 以内。

## 六、消融实验

为验证各模块的真实贡献，我们依次移除单个模块后测试：

| 移除的模块 | 成功率变化 | 结论 |
|------------|:---:|------|
| 无（完整 V10） | 78.7% | 基线 |
| -ActionFilter | 78.7% → 52.3% | **最重要模块**，非法动作暴增 |
| -ReflectionEngine | 78.7% → 61.5% | 失败后无法恢复 |
| -TaskDecomposer | 78.7% → 65.2% | 长程任务（Move Rope, Arrange Cabinet）大幅下降 |
| -RoleAssigner | 78.7% → 71.8% | 碰撞冲突增加 |
| -FormatHealer | 78.7% → 74.1% | 格式错误导致随机失败 |
| -LoopDetector | 78.7% → 75.5% | 偶发死循环导致超时 |
| -ExperienceMemory | 78.7% → 76.2% | 初期成功率降低，随着记忆积累恢复 |

**结论**：ActionFilter 和 ReflectionEngine 是贡献最大的两个模块，合计贡献约 26 个百分点的提升。

## 八、总结与展望

### 8.1 项目总结

本项目提出了一套**纯 Prompt Engineering + API 调用**的多机器人协同规划方案，在不进行任何模型训练的情况下，将 RocoBench 6 个任务的平均成功率从基线的 25.8% 提升至 78.7%。

核心方法论：**LLM 只负责意图选择，系统负责可行性保障。**

### 8.2 关键收获

1. **动作过滤 > Prompt 优化**：限制 LLM 的输出空间远比优化 Prompt 更有效
2. **失败恢复 > 一次规划**：机器人操作中 30% 的动作会失败，恢复能力是关键
3. **模块化 > 单一大 Prompt**：多个小模块各自解决特定问题，比一个"万能 Prompt"更可控
4. **确定性 > 创造性**：机器人控制需要稳定输出，不是"有创意的文本"

### 8.3 未来工作

1. 实现"负向 Prompt"机制（禁止动作黑名单）
2. 训练动作翻译小模型实现毫秒级推理
3. 各任务独立调优 temperature
