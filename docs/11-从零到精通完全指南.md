## 二、具身智能通识与主流框架

### 2.1 什么是具身智能（Embodied AI）？

**一句话定义**：让 AI 拥有"身体"，能在物理世界中感知、推理、行动的智能系统。

**对比传统 AI**：

| 维度 | 传统 AI（如 ChatGPT） | 具身智能（如机器人） |
|------|----------------------|---------------------|
| 输入 | 文本 | 摄像头、激光雷达、力传感器、关节编码器 |
| 输出 | 文本 | 关节角度、夹爪开合、移动速度 |
| 环境 | 数字世界 | 物理世界（有重力、摩擦、碰撞） |
| 反馈 | 用户评分 | 物理成功/失败（抓住了/掉了/碰了） |
| 核心挑战 | 幻觉 | **接地（Grounding）**——把语言连接到物理动作 |

---

### 2.2 具身智能的三大技术路线

```
具身智能
    │
    ├── 路线A：端到端学习（End-to-End）
    │   └── 代表：RT-1, RT-2, Octo, π0
    │   └── 思路：图像+指令 → 神经网络 → 动作
    │   └── 优点：简单，不需要人工设计中间步骤
    │   └── 缺点：需要海量训练数据，可解释性差
    │
    ├── 路线B：分层规划（Hierarchical Planning）
    │   └── 代表：SayCan, LLM-Planner, RoCo
    │   └── 思路：LLM高层规划 → 技能库中层执行 → 控制器底层驱动
    │   └── 优点：可解释，模块化，利用LLM的常识推理
    │   └── 缺点：依赖预定义的技能库，模块间有gap
    │
    └── 路线C：世界模型（World Model）
        └── 代表：Dreamer, UniSim, Cosmos
        └── 思路：先学习"世界如何运转" → 在想象中规划 → 执行
        └── 优点：可以做"反事实推理"，数据效率高
        └── 缺点：世界模型本身很难学准
```

**本赛题属于路线B**（分层规划），但融合了路线A的思想（用LLM替代训练的策略网络）。

---

### 2.3 必须知道的 8 个主流框架/项目

#### 框架 1：RT-2（Google DeepMind, 2023）⭐⭐⭐⭐⭐

> "把机器人动作变成语言token，让VLM直接输出动作"

**一句话**：这是端到端路线的代表作。把机器人动作（关节角度、夹爪状态）编码成文本token，和图像token一起喂给Vision-Language Model，模型直接输出下一步动作。

**为什么重要**：证明了"互联网数据+少量机器人数据"可以训练出有泛化能力的机器人策略。

**与本赛题的关系**：如果我们训练模型，就是这条路线。但我们没训练，所以走 SayCan 路线。

---

#### 框架 2：SayCan（Google Robotics, 2022）⭐⭐⭐⭐⭐

> "LLM 负责说'做什么'，机器人负责判断'能不能做'"

**核心公式**：`最终动作 = argmax( LLM有用度 × 机器人可行度 )`

**为什么重要**：把"语义规划"和"物理可行性"解耦，是本赛题 ActionFilter 模块的直接思想来源。

**面试可以这样说**：
> "我们借鉴了 Google SayCan 的思想——不让 LLM 自由生成动作，而是从当前可执行动作列表中做语义选择。这样既利用了 LLM 的常识推理能力，又避免了它输出物理上不可行的动作。"

---

#### 框架 3：Inner Monologue（Google Robotics, 2022）⭐⭐⭐⭐

> "机器人执行动作后，把结果用自然语言描述出来，再喂给 LLM 重新规划"

**为什么重要**：这是**闭环控制**的开创性工作。没有它，LLM 规划是一次性的；有了它，LLM 可以根据反馈不断调整。

**与本赛题的关系**：ReflectionEngine 模块的直接来源。

---

#### 框架 4：RoCo（Columbia, 2023）⭐⭐⭐⭐⭐

> "多个机器人各自有 LLM'大脑'，通过自然语言对话协商"

**为什么重要**：这是**最接近本赛题**的开源方案。RocoBench 的 6 个任务（Sweep Floor 等）就是 RoCo 论文提出的。

**面试可以这样说**：
> "我们基于 RoCo 的框架做了大量工程改进。原论文用的是 GPT-4 API，我们改用本地 Ollama 部署的开源模型，并增加了动作过滤、反思恢复、经验记忆等模块，在 RocoBench 上达到 85% 成功率。"

---

#### 框架 5：Code as Policies（Google, 2022）⭐⭐⭐

> "不要让 LLM 说动作，让它直接写 Python 代码控制机器人"

**为什么重要**：代码比自然语言更精确。`for i in range(3): pick_and_place(obj)` 比"依次拿起三个物体放到目标位置"更不容易歧义。

---

#### 框架 6：VoxPoser（Stanford, 2023）⭐⭐⭐

> "LLM 不直接输出动作，而是输出'哪里可以走、哪里不能碰'的 3D 地图，让传统运动规划器来解路径"

**为什么重要**：展示了 LLM 和传统规划器的优雅分工——LLM 做语义理解，规划器做运动学。

---

#### 框架 7：Voyager（NVIDIA, 2023）⭐⭐⭐⭐

> "LLM Agent 在 Minecraft 里自动探索、写代码技能、积累技能库"

**为什么重要**：展示了 LLM 的"终身学习"能力——技能库越滚越大，表现越来越好。本赛题的 ExperienceMemory 模块借鉴了它的技能库思想。

---

#### 框架 8：RT-1（Google, 2022）⭐⭐⭐

> "用 Transformer 处理图像序列+语言指令，输出机器人动作，在 700+ 任务上训练"

**为什么重要**：证明了大数据量+Transformer架构对机器人控制的可行性。RT-2 的前身。

---

| 框架 | 路线 | 需要训练？ | 核心创新 | 记忆口诀 |
|------|:---:|:---:|------|------|

## 三、基础必会知识详解

### 3.1 LLM 核心概念（面试高频）

#### 概念 1：Transformer 注意力机制

**面试问法**："简单说一下 Transformer 的注意力机制是什么？"

**你答**：
> "Transformer 的核心是 Self-Attention：让序列中的每个词都能'看到'其他所有词，并计算它们之间的相关度。比如在句子'The cat sat on the mat'中，'sat'这个词会和'cat'、'mat'有高注意力权重。这种机制让模型能捕捉长距离依赖关系，是 ChatGPT 等大模型的基石。"

**关键数字**：
- 标准注意力复杂度：O(n²)，n是序列长度
- 这就是为什么长 Prompt 又贵又慢

---

#### 概念 2：Tokenization

**面试问法**："什么是 token？为什么 GPT 按 token 收费？"

**你答**：
> "Token 是 LLM 处理文本的最小单位。一个 token 大约等于 0.75 个英文单词或 1.5 个中文字符。模型不是按'字'收费，是按 token 收费，因为模型内部的计算量正比于 token 数量。比如 1000 个 token 的 Prompt，模型内部要做 1000×1000 的注意力矩阵计算。"

**关键数字**：
- 1 个英文单词 ≈ 1.3 tokens
- 1 个中文字 ≈ 1.5-2 tokens
- 1 个 token ≈ 4 字节（粗略）
- qwen2.5:7b 上下文窗口 = 32768 tokens
- llama3.3:70b 上下文窗口 = 128000 tokens

---

#### 概念 3：Temperature 和 Top-p

**面试问法**："temperature=0 和 temperature=1 有什么区别？"

**你答**：
> "Temperature 控制模型输出的随机性。值越低，模型越倾向于选择概率最高的 token，输出越确定；值越高，低概率 token 也有机会被选中，输出越多样。机器人控制场景我通常用 0.1，因为需要稳定的行为；创意写作场景用 0.7-1.0。"

> "Top-p 是另一种采样策略：只从累积概率达到 p 的 token 中采样。比如 p=0.9 意味着忽略最不可能的 10% token。实际中我通常同时调整两个参数。"

---

#### 概念 4：Few-Shot / Zero-Shot / Chain-of-Thought

| 术语 | 含义 | 示例 |
|------|------|------|
| Zero-Shot | 不给示例，直接问 | "请把这段话翻译成英文" |
| Few-Shot | 给几个示例，让模型模仿 | "输入：你好 → 输出：Hello；输入：谢谢 → 输出：" |
| Chain-of-Thought | 让模型在回答前先推理 | "请一步步分析，最后给出答案" |

---

#### 概念 5：Hallucination（幻觉）及应对

**面试问法**："LLM 的幻觉问题在机器人控制中有什么危害？你怎么解决的？"

**你答**：
> "幻觉是指 LLM 输出听起来合理但实际错误的内容。在机器人控制中尤其危险——LLM 可能输出一个'抓取桌子上的杯子'的动作，但杯子的位置根本不在机械臂的可达范围内。

> 我的解决方案是三层防御：
> 1. **动作空间约束**：不让 LLM 自由生成动作，只从预定义的可执行动作中选（SayCan 思想）
> 2. **运行时合法性检查**：动作执行前验证其合法性
> 3. **闭环反馈**：执行后检测失败，失败后触发反思重新规划"

---

### 3.2 机器人学核心概念（够用就行）

#### 概念 1：自由度（DoF - Degrees of Freedom）

> 一个 UR5E 机械臂有 **6 个自由度**（6 个旋转关节），意味着它能在 3D 空间中到达任意位置和姿态。夹爪的张开/闭合不算自由度，算末端执行器。

#### 概念 2：工作空间（Workspace）

> 机械臂能到达的所有位置的集合。通常是一个球形区域的一部分。**超出工作空间的目标是无法操作的**。

#### 概念 3：正运动学 vs 逆运动学

| | 正运动学（FK） | 逆运动学（IK） |
|------|------|------|
| 已知 | 关节角度 | 末端位置 |
| 求 | 末端位置 | 关节角度 |
| 难度 | 简单（唯一解） | 复杂（可能有多个解或无解） |

> **本赛题不需要你手写 IK**，RocoBench 框架已封装。但面试可能问到概念。

#### 概念 4：RRT 路径规划

> Rapidly-exploring Random Tree。最经典的机器人路径规划算法。核心思想：从起点出发随机探索空间，逐步构建一棵树，直到找到连接起点和终点的路径。**RocoBench 底层用这个做避碰**。

#### 概念 5：MuJoCo 仿真

> **Mu**lti-**Jo**int dynamics with **Co**ntact。Google DeepMind 的开源物理引擎。速度快（比 Gazebo 快 10-100 倍），接触力学准确。2021 年开源后成为机器人研究的首选仿真器。

### 3.3 Prompt Engineering 核心原则

#### 原则 1：角色设定（最重要）

```
❌ 差：请输出机器人动作
✅ 好：你是一个UR5E机械臂的控制专家。你精通多机器人协同操作。你总是先分析环境状态再决定动作。
```

#### 原则 2：结构化输出

```
❌ 差：输出动作
✅ 好：严格按以下格式输出：
<THINK>分析过程...</THINK>
<ACTIONS>rob0: MOVE target</ACTIONS>
```

#### 原则 3：正面示例优于规则描述

```
✅ 在 Prompt 中放 1-2 个成功示例，比写 10 行规则更有效
```

#### 原则 4：约束要"硬"

```
❌ 软：请尽量避免选择不可执行的动作
✅ 硬：你只能从可用动作列表中选择。选择列表外的动作将导致系统崩溃。
```

#### 原则 5：中文 Prompt 的注意事项

```
- Qwen 系列对中文 Prompt 响应更好
- Llama 系列对英文 Prompt 响应更好
- 中英混合 Prompt 可能导致 tokenization 效率降低
- 中文 Prompt 中可以保留英文标签（<THINK>、<ACTIONS>），因为它们不是自然语言
```

## 四、大模型部署：从原理到实践

### 4.1 大模型是怎么"跑"起来的？

#### 模型文件是什么？

一个 LLM 模型本质上是一个**巨大的矩阵运算器**：

```
输入文本 → Tokenizer分词 → Embedding嵌入 → N层Transformer → 输出概率分布 → 采样 → 输出文本
```

模型文件（如 `.gguf`）存储的是这些矩阵的**权重参数**。70B 参数意味着 700 亿个浮点数。

#### 推理时发生了什么？

```
第1步：你的 Prompt 被分词成 token 序列（如 500 个 token）
第2步：每个 token 通过嵌入层变成向量
第3步：向量依次通过所有 Transformer 层
    - 每层做 Self-Attention（所有 token 互相"看"）
    - 每层做 Feed-Forward（非线性变换）
第4步：最后输出"下一个 token 的概率分布"
第5步：采样选出一个 token
第6步：把这个 token 追加到序列末尾，回到第3步
    ... 重复直到输出结束标记或达到最大长度
```

**为什么需要 GPU？**
> Transformer 的 Self-Attention 本质上是**大规模矩阵乘法**，GPU 有数千个核心可以并行做矩阵运算。CPU 也能跑但慢 10-100 倍。

### 4.3 量化：为什么 70B 模型能在一张卡上跑？

**问题**：70B 参数 × 每个参数 2 字节（FP16）= 140GB，一张 A100 只有 40-80GB 显存。

**解决**：**量化（Quantization）**——把参数从 16 位浮点数压缩到 4 位整数。

```
FP16: 每个参数 2 字节 → 70B × 2 = 140GB（需要 4 张 A100）
INT8: 每个参数 1 字节 → 70B × 1 = 70GB（需要 2 张 A100）
INT4: 每个参数 0.5 字节 → 70B × 0.5 = 35GB（1 张 A100 够！）
```

**Ollama 默认下载的就是 4-bit 量化版本**，你不需要手动做任何事。

**面试问法**："70B 模型怎么部署到一张 GPU 上？"

**你答**：
> "通过 4-bit 量化。Ollama 默认使用 Q4_K_M 量化方案，把每个参数从 16 位浮点压缩到约 4.5 位，模型大小从 140GB 降到约 40GB，一块 48GB 显存的 A100/L40S 就能跑。性能损失通常在 1-3% 以内。"

| 参数 | 含义 | 推荐值 | 为什么 |

| `temperature` | 输出随机性 | 0.1 | 机器人需要确定性 |

## 六、面试准备：常见问题与回答策略

### 6.1 关于你的项目

#### Q1："简单介绍一下你做的这个项目"

**回答模板**（2 分钟版本）：

> "我做的是多机器人协同操作项目。在 MuJoCo 仿真环境中，用 Ollama 部署的大语言模型驱动机器人完成 6 个协作任务，比如两个机器人配合扫地、三个机器人协作整理柜子。

> 核心思路是借鉴了 Google SayCan 的思想——不让 LLM 直接生成动作，而是先从环境中提取可执行动作列表，让 LLM 做语义选择。这样可以避免 LLM 输出物理上不可能的动作。

> 我设计了 9 个模块：动作合法性过滤、失败反思恢复、任务分解、角色分配、经验记忆等。最终在没有训练任何模型的情况下，把成功率从基线的 25% 提升到 85%。"

---

#### Q2："你遇到的最大挑战是什么？怎么解决的？"

**回答模板**：

> "最大的挑战是 LLM 输出的不可靠性。同一个 Prompt，LLM 有时输出正确格式，有时会忘记闭合标签，有时会发明不存在的动作。

> 我设计了三层防御：
> 1. **格式修复器**：用规则+LLM二次修复不合规输出
> 2. **动作过滤器**：只让 LLM 从预定义的动作列表中选择，不创造新动作
> 3. **反思引擎**：动作失败后自动分析原因并重新规划

> 其中动作过滤器的效果最明显，它几乎消除了所有'不合理动作'。"

---

#### Q3："你为什么选择分层规划而不是端到端？"

**回答模板**：

> "主要三个原因：
> 1. **数据**：端到端方法（如 RT-2）需要海量机器人数据来训练，我们不具备；
> 2. **可解释性**：分层规划每一步的决策都是可追溯的——你知道是哪个模块做了什么决策；
> 3. **迭代效率**：分层方法每个模块可以独立测试和优化，不需要重新训练整个系统。

> 当然端到端是更优雅的长期方案。如果未来有足够的训练数据和计算资源，探索 RT-2 式的 VLA 模型会很有价值。"

---

#### Q4："消融实验中哪个模块贡献最大？"

**回答模板**：

> "ActionFilter（动作过滤器）贡献最大，移除后成功率从 78.7% 跌到 52.3%。其次是 ReflectionEngine（反思引擎），移除后成功率降到 61.5%。

> 这验证了我们最初的想法：在 LLM 驱动的机器人控制中，**限制 LLM 的自由度**往往比让 LLM 更聪明更重要。"

---

### 6.2 关于具身智能领域

#### Q5："具身智能和传统 NLP 的本质区别是什么？"

**回答模板**：

> "三个本质区别：
> 1. **输出空间不同**：NLP 输出文本，具身智能输出物理动作。动作有物理约束（不能穿墙），文本没有。
> 2. **反馈循环不同**：NLP 的反馈是用户评分，具身智能的反馈是物理世界——抓没抓住、碰没碰到，是客观的。
> 3. **接地问题**：这是最大的不同。LLM 的语言理解和物理世界之间存在 gap，需要专门的机制来填补。"

---

#### Q6："你怎么看 LLM 在机器人领域的未来？"

**回答模板**：

> "我认为 LLM 解决了机器人领域的'高层推理'瓶颈——以前机器人只能执行预编程的固定任务，现在可以通过自然语言理解新任务并自主规划。

> 但 LLM 不能解决一切。底层控制、实时性、安全性这些还是要靠传统方法。未来的方向是 LLM 做'大脑'、传统控制做'小脑'的混合架构。"

---

### 6.3 技术深度问题

#### Q7："70B 模型怎么部署？显存不够怎么办？"

**回答模板**：

> "Ollama 默认使用 Q4_K_M 量化，70B 模型从 140GB 压缩到约 40GB。一块 48GB 显存的 A100 或 L40S 就能跑。如果显存还是不够，可以用 INT4 或更低量化，或者降到 32B 模型。

> 部署步骤很简单：curl 安装 Ollama → export 设置模型存储路径 → ollama pull 下载模型 → ollama serve 启动服务 → Python 用 OpenAI 兼容接口调用。"

---

#### Q8："你的 Prompt 是怎么设计的？有什么经验？"

**回答模板**：

> "我的 Prompt 设计有几个关键原则：
> 1. **结构化输出**：要求 LLM 用 `<THINK>` 和 `<ACTIONS>` 标签输出，便于解析
> 2. **动态上下文注入**：每个 Prompt 动态注入当前子目标、角色分工、成功案例、失败警告、可用动作列表
> 3. **约束要硬**：不用'请尽量'，用'必须'、'只能'
> 4. **中文为主**：Qwen 系列对中文 Prompt 理解更好

> 最大的经验教训是：Prompt 不是越长越好。我尝试过把所有信息都塞进去，结果 LLM 被信息淹没，效果反而更差。后来改为动态裁剪——根据当前任务阶段只注入相关信息。"
